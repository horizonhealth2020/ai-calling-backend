<objective>
## Goal
Add 48-hour stale alert system with a "My Queue" personal view for CS reps and an owner/admin stale overview across all agents.

## Purpose
CS agents need to know which records need urgent attention. Pending terms must be resolved within 48 hours; chargebacks must be worked (attempt logged) within 48 hours. Owners need accountability visibility into which agents have stale records.

## Output
- API: GET /api/stale-summary endpoint returning stale counts per agent + stale records
- UI: "My Queue" tab for CUSTOMER_SERVICE users showing their assigned records with stale highlighting
- UI: Owner/admin stale overview section on CS tracking showing per-agent stale counts
</objective>

## AC-1: Stale summary API endpoint
```gherkin
Given 3 unresolved chargebacks assigned to "Alex" (2 stale, 1 fresh) and 2 unresolved pending terms assigned to "Alex" (1 stale, 1 fresh)
When GET /api/stale-summary is called
Then the response includes { agents: [{ name: "Alex", staleChargebacks: 2, stalePendingTerms: 1, totalStale: 3 }], records: [...] }
And stale is defined as: pending terms > 48h from midnight of createdAt without resolution, chargebacks > 48h from midnight of createdAt without any contact attempt (or > 48h since last attempt)
```

## AC-2: Stale summary filterable by agent name
```gherkin

```

## AC-3: My Queue tab visible to CS reps

## AC-4: My Queue shows stale items prioritized
```gherkin
Given Alex has 5 assigned chargebacks (2 stale) and 3 assigned pending terms (1 stale)
When My Queue loads
Then stale items appear at the top with a red "STALE" badge and time-since-stale indicator
And fresh items appear below without the badge
```

## AC-5: Pending term staleness — resolution only
```gherkin
Given a pending term assigned to Alex, created 50 hours ago, with 2 logged attempts but not resolved
When My Queue loads
Then the pending term shows as STALE (attempts don't clear staleness for pending terms)
```

## AC-6: Chargeback staleness — resets on attempt
```gherkin
Given a chargeback assigned to Alex, created 50 hours ago, with a contact attempt logged 10 hours ago
When My Queue loads
Then the chargeback is NOT stale (last attempt was < 48 hours ago)

Given a chargeback assigned to Alex, created 100 hours ago, with last attempt 50 hours ago
When My Queue loads
Then the chargeback IS stale (last attempt was > 48 hours ago)
```

## AC-7: Owner/admin stale overview
```gherkin
Given the logged-in user has OWNER_VIEW or SUPER_ADMIN role
When they view the CS tracking page
Then a "Stale Overview" card appears at the top showing per-agent stale counts
And each agent row shows: name, stale chargebacks count, stale pending terms count, total stale
And agents with 0 stale items are hidden or dimmed
```

## AC-8: Already-resolved records never stale
```gherkin
Given a pending term resolved 60 hours after creation
When stale summary is computed
Then it does NOT appear as stale (resolved records are excluded)
```

  <name>Task 1: Stale summary API endpoint</name>
  <files>apps/ops-api/src/routes/chargebacks.ts, apps/ops-api/src/routes/pending-terms.ts</files>
  <action>
    Add a new GET /api/stale-summary endpoint in chargebacks.ts (since it spans both record types):

    **Staleness calculation (server-side SQL/Prisma):** <!-- audit-strengthened -->
    - Deadline = midnight UTC of createdAt date + 48 hours (use UTC consistently — this project stores timestamps in UTC via Prisma/Postgres default, and "morning" = start of UTC day to avoid timezone ambiguity)
    - For pending terms: stale if `now > deadline AND resolvedAt IS NULL`
    - For chargebacks: stale if `resolvedAt IS NULL AND (no ContactAttempts exist AND now > deadline) OR (latest ContactAttempt.createdAt + 48h < now)`

    **Implementation:**
    - Role: requireRole("CUSTOMER_SERVICE", "SUPER_ADMIN", "OWNER_VIEW")
    - Optional query param: `assignedTo` (string, case-insensitive filter)
    - Query unresolved chargebacks where stale: <!-- audit-strengthened: batch query, not N+1 -->
      ```
      1. Find all unresolved chargebacks (resolvedAt IS NULL)
      2. LEFT JOIN to get MAX(ContactAttempt.createdAt) per chargeback in a single query
         Use Prisma raw query or groupBy to avoid N+1:
         SELECT cb.*, MAX(ca.created_at) as last_attempt_at
         FROM chargeback_submissions cb
         LEFT JOIN contact_attempts ca ON ca.chargeback_submission_id = cb.id
         WHERE cb.resolved_at IS NULL
         GROUP BY cb.id
      3. For each row:
         a. If last_attempt_at IS NULL: stale if now > (midnight UTC of cb.created_at + 48h)
         b. If last_attempt_at exists: stale if now > (last_attempt_at + 48h)
      ```
    - Query unresolved pending terms where stale:
      ```
      1. Find all unresolved pending terms (resolvedAt IS NULL)
      2. Stale if now > (midnight of pt.createdAt + 48h)
      ```
    - If assignedTo param: filter both queries by assignedTo (case-insensitive, trimmed — use `.toLowerCase().trim()` on both sides) <!-- audit-added -->
    - Group by assignedTo to build per-agent summary
    - Return: `{ agents: [{ name, staleChargebacks, stalePendingTerms, totalStale }], records: { chargebacks: [...], pendingTerms: [...] } }`
    - Each record in the response includes: id, memberName/memberCompany, assignedTo, createdAt, type ("chargeback"/"pending_term"), staleSince (deadline timestamp), lastAttemptAt (for chargebacks)
    - **Also return all unresolved assigned records (not just stale)** so My Queue can show fresh items without separate fetches. Add `allRecords: { chargebacks: [...], pendingTerms: [...] }` containing all unresolved records for the filtered agent. Stale records appear in both `records` and `allRecords`. <!-- audit-added -->

    **Performance note:** Use raw SQL or Prisma $queryRaw for the chargeback LEFT JOIN + MAX pattern. Avoid N+1 (fetching attempts per chargeback in a loop).
  </action>
  <verify>TypeScript compiles. Endpoint returns correct stale/fresh classification based on 48h rule with different logic for chargebacks vs pending terms.</verify>
  <done>AC-1 satisfied: stale summary endpoint. AC-2 satisfied: filterable by assignedTo. AC-5 satisfied: PT staleness ignores attempts. AC-6 satisfied: CB staleness resets on attempt. AC-8 satisfied: resolved records excluded.</done>
