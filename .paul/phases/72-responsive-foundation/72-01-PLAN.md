<objective>
## Goal
Ship the responsive foundation that Phases 73-76 consume: (a) formalize breakpoint tokens as a shared JS/CSS source of truth, (b) add `useBreakpoint` hook + `MobileDrawer` primitive in `@ops/ui`, (c) extend `responsive.css` with the missing utilities the mobile retrofit needs (touch targets, bottom-sheet base, mobile typography scale), (d) replace the dashboard's hover-to-expand nav with a proper hamburger + drawer on mobile while preserving the current collapse behavior on desktop.

## Purpose
Every subsequent mobile phase (73 Manager, 74 Payroll, 75 Owner, 76 CS) needs the same three things: a JS-side way to ask "am I on mobile?" for conditional render (table→card, inline-row→bottom-sheet), CSS utilities that stack/hide/size content per breakpoint, and a drawer component for nav + sidebars. Building them once in `@ops/ui` avoids five divergent implementations and keeps the inline-CSSProperties constraint intact.

## Output
- `breakpoints` export in `tokens.ts` (single source of truth, matches existing CSS media-query values)
- `useBreakpoint()` / `useIsMobile()` hook under `@ops/ui`
- `MobileDrawer` component (slide-over from left/right/bottom) under `@ops/ui`
- Extended `responsive.css` utilities: `.touch-target` (min 44px height), `.bottom-sheet-base`, `.mobile-text-sm/base/lg` (type scale that scales up on mobile for readability), `.stack-mobile-md` (stack with larger gap)
- Dashboard layout nav refactored: hamburger + `MobileDrawer` below 768px, hover-collapse preserved ≥ 768px
</objective>

## AC-3: MobileDrawer primitive — accessible and production-safe
```gherkin

And Tab cycles are trapped inside the drawer until it closes
And on close, focus is restored to the element that was focused when the drawer opened
And `role="dialog"`, `aria-modal="true"`, and `aria-label` (from required prop) are present

Given the drawer is open
Then body scroll is locked (prior `document.body.style.overflow` captured and restored on close)

Given `ariaLabel` is omitted at the type level
Then TypeScript errors at build time (required prop — not optional)
```

    Add to `tokens.ts` after the `typography` export:
    ```ts
    /**
     * Breakpoint boundaries in pixels. Keys are deliberately explicit about
     * whether the number is a max-width (ceiling) or min-width (floor) to
     * prevent off-by-one bugs in consumer code.
     *
     *   mobile  : width <= mobileMax
     *   tablet  : tabletMin <= width <= tabletMax
     *   desktop : width >= desktopMin
     *   wide    : width >= wideMin
     */
    export const breakpoints = {
      mobileMax: 767,
      tabletMin: 768,
      tabletMax: 1023,
      desktopMin: 1024,
      wideMin: 1440,
    } as const;

    export const mediaQuery = {
      mobile: `(max-width: ${breakpoints.mobileMax}px)`,
      tablet: `(min-width: ${breakpoints.tabletMin}px) and (max-width: ${breakpoints.tabletMax}px)`,
      desktop: `(min-width: ${breakpoints.desktopMin}px)`,
      wide: `(min-width: ${breakpoints.wideMin}px)`,
    } as const;
    ```
    Re-export `breakpoints` and `mediaQuery` from `index.tsx` alongside existing `colors`, `spacing`, etc.

    Extend `responsive.css` WITHIN the existing media-query blocks (do not introduce new blocks — preserves current structure):
    - Inside `@media (max-width: 767px)` add:
      - `.touch-target { min-height: 44px; min-width: 44px; }`
      - `.mobile-text-base { font-size: 14px !important; line-height: 1.5 !important; }`
      - `.mobile-text-lg { font-size: 16px !important; line-height: 1.5 !important; }`
      - `.stack-mobile-md { flex-direction: column !important; gap: 16px !important; }`

      ```css
        border-top: 1px solid var(--border-default);
        border-radius: var(--radius-2xl) var(--radius-2xl) 0 0;
        box-shadow: var(--shadow-lg);
        z-index: 1000;
      }
      ```

        ```tsx
        return <MobileNav />;
        ```

      Handle Safari < 14 via `addListener` / `removeListener` fallback if typeof addEventListener === 'undefined' on the MediaQueryList.

    Create `packages/ui/src/components/MobileDrawer.tsx`:
    <!-- audit-added: ariaLabel required (not optional); close button; focus trap; scroll-lock correctness; reduced-motion -->
    - Props (note `ariaLabel` is REQUIRED):
      ```ts
      type MobileDrawerProps = {
        open: boolean;
        onClose: () => void;
        ariaLabel: string;           // REQUIRED — every dialog must be named
        side?: "left" | "right" | "bottom";
        children: React.ReactNode;
        id?: string;                 // for aria-controls wiring from the opener
      };
      ```
    - Returns `null` when `!open`.
    - Renders: backdrop (fixed inset, rgba(0,0,0,0.5), click → onClose) + panel (fixed, slides in from `side`).
    - `side="left"` / `"right"`: 85vw max 360px, full height.
    - `side="bottom"`: full width, max 85vh, uses `.bottom-sheet-base` class.
    - Transition: `transform` 200ms `ease-out` on open, 150ms on close.
      Respect reduced motion — if `window.matchMedia("(prefers-reduced-motion: reduce)").matches`,
      set `transition: "none"` instead.
    - Every drawer renders an in-drawer close button (×) at the top-right corner, `aria-label="Close"`,
      `className="touch-target"`. Closing via any of: backdrop tap, ESC, this button, or programmatic `onClose`.

    - **Focus management** (required):
      - On open: capture `document.activeElement` in a ref (`previouslyFocusedRef`). Move focus
        to the first focusable element inside the drawer; if none, focus the Close button.
      - While open: trap Tab/Shift-Tab within the drawer. Implementation: on Tab keydown, query all
        focusable descendants (`button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])`)
        and cycle focus at the boundaries.
      - On close: restore focus to `previouslyFocusedRef.current` if still in the DOM and focusable;
        otherwise fall through (do NOT throw).

    - **Body scroll lock** (correctness required):
      - On open: capture `document.body.style.overflow` into a local variable, then set it to `"hidden"`.
      - On close (effect cleanup): restore the captured prior value — do NOT hardcode `"auto"` or `""`,
        because a parent modal could have already set it.
      - Wrap in try/finally equivalent (React cleanup runs on unmount even if render throws, so using
        `useEffect` cleanup is sufficient — but the captured value must live outside the effect body
        in a ref so cleanup sees it).

    - ESC handler: `useEffect` + `window.addEventListener("keydown", ...)`; cleans up on close.
    - `role="dialog"`, `aria-modal="true"`, `aria-label={ariaLabel}` (always — prop is required).
    - Inline `React.CSSProperties` styling using design tokens (`colors.bgSurface`, `radius.xl`, `spacing`, `motion`).
    - DO NOT import the existing ConfirmModal — independent component, similar patterns (but this one has
      focus trap and scroll-lock that ConfirmModal does not ship today, so they cannot be shared yet).
    - Add to `packages/ui/src/components/index.ts` barrel export, and re-export from `packages/ui/src/index.tsx`.

    Avoid: portals (consistent with ConfirmModal precedent — z-index is enough); wrapper libraries
    (`react-focus-lock`, `body-scroll-lock`, `framer-motion`) — zero new runtime dependencies for this phase;
    importing Tailwind or emotion.
