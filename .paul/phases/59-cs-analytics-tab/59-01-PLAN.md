<objective>
## Goal
Replace the Resolved Log tab with a CS Analytics tab — rep performance metrics, chargeback resolution patterns, pending term category breakdown, and rep drill-down — restricted to OWNER_VIEW and SUPER_ADMIN.

## Purpose
Owners need to see CS team effectiveness at a glance — who resolves fastest, what chargeback patterns look like, which holdReasons dominate pending terms. The current Resolved Log is a flat audit trail. Analytics transforms that data into actionable insights for coaching reps and spotting process issues.

## Output
- New `/api/cs/analytics` endpoint returning aggregated CS metrics
- CSAnalytics.tsx component with 4 sections (rep performance, chargeback patterns, pending term categories, rep drill-down)
- Resolved Log tab renamed to "Analytics" in CS page navigation
- Reuses Recharts (already installed in Phase 58)
</objective>

## AC-1: CS Analytics API returns aggregated metrics
```gherkin
Given an authenticated OWNER_VIEW or SUPER_ADMIN user
When they call GET /api/cs/analytics?range=month (or from/to params)
Then the response contains:
  - repPerformance: array of { repName, chargebackCount, pendingTermCount, resolvedCount, resolutionRate, avgTurnaroundHours } per assignedTo rep
  - chargebackPatterns: { matchStatusDistribution: { MATCHED, UNMATCHED, MULTIPLE }, resolutionTypeDistribution: { recovered, closed, unresolved } }
  - pendingTermCategories: { holdReasonDistribution: { [reason]: count }, resolutionTypeDistribution: { saved, cancelled, unresolved } }
  - totals: { totalChargebacks, totalPendingTerms, totalResolved, overallResolutionRate, avgTurnaroundHours }
And all numeric values are serialized as numbers (Decimal fields converted via Number())
And turnaround hours are rounded to 1 decimal place
```

  <name>Task 1: CS Analytics API endpoint and aggregation service</name>
  <files>apps/ops-api/src/services/csAnalyticsAggregator.ts, apps/ops-api/src/routes/cs-reps.ts</files>
  <action>
    Create new service `csAnalyticsAggregator.ts` with function `getCsAnalytics(dateRange)`:

    **dateRange helper returns `{ gte: Date, lt: Date }` — use these field names in Prisma where clauses.**

    1. **repPerformance** — Query ChargebackSubmission and PendingTerm grouped by `assignedTo` within date range.
       For each rep:
       - chargebackCount: count of chargebacks assigned to this rep
       - pendingTermCount: count of pending terms assigned to this rep
       - resolvedCount: count where resolvedAt is not null
       - resolutionRate: resolvedCount / (chargebackCount + pendingTermCount), guard div-by-zero
       - avgTurnaroundHours: avg of (resolvedAt - submittedAt) in hours for resolved items, guard div-by-zero
       Exclude reps where assignedTo is null. Convert Decimal fields to Number.

    2. **chargebackPatterns** — Query ChargebackSubmission within date range:
       - matchStatusDistribution: groupBy matchStatus, count per status. Null matchStatus → "UNKNOWN"
       - resolutionTypeDistribution: count by resolutionType (recovered/closed). Items with null resolvedAt → "unresolved"

    3. **pendingTermCategories** — Query PendingTerm within date range:
       - holdReasonDistribution: groupBy holdReason, count per reason. Null/empty → "No Reason"
       - resolutionTypeDistribution: count by resolutionType (saved/cancelled). Items with null resolvedAt → "unresolved"

    4. **totals** — Aggregate from above: totalChargebacks, totalPendingTerms, totalResolved, overallResolutionRate, avgTurnaroundHours

    Use Promise.all for parallel queries. Wrap in try/catch per sub-query for partial failure resilience (matching Phase 58 pattern).

    **Response caching:** Set `Cache-Control: private, max-age=60` header on the analytics response — trend data is historical and changes slowly, so a 60s client-side cache reduces redundant fetches when switching tabs. <!-- deferred-added -->

    Add route in cs-reps.ts:
    ```
    GET /api/cs/analytics?range=[preset]|from=&to=
    Roles: OWNER_VIEW, SUPER_ADMIN (via requireAuth + requireRole)
    ```
    Validate with dateRangeQuerySchema, return 400 if dateRange returns undefined.

    **Also add drill-down endpoint:**
    ```
    GET /api/cs/analytics/rep/:repName?range=[preset]|from=&to=
    Roles: OWNER_VIEW, SUPER_ADMIN
    ```
    Returns array of { type: "chargeback"|"pending_term", memberName, resolvedAt, resolutionType, resolutionNote, originalAmount } for the specified rep, sorted by resolvedAt desc. **Support pagination: accept `?limit=50&offset=0` query params (default limit=50, max=200).** Decode repName with decodeURIComponent. **Use case-insensitive matching on assignedTo (mode: "insensitive") since assignedTo is a free-text field with potential casing inconsistencies.** <!-- audit-added --> Also return 400 if dateRange returns undefined. <!-- audit-added -->
  </action>
  <verify>
    curl -H "Authorization: Bearer $TOKEN" "http://localhost:8080/api/cs/analytics?range=month" returns 200 with repPerformance, chargebackPatterns, pendingTermCategories, totals.
    curl -H "Authorization: Bearer $TOKEN" "http://localhost:8080/api/cs/analytics/rep/RepName?range=month" returns 200 with array of resolved items.
  </verify>
