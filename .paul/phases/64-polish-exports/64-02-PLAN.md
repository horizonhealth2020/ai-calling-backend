<task type="auto">
  <name>Task 1: Fix implicit any in all ops-api files</name>
  <files>All 16 files listed in files_modified</files>
  <action>
    For each file, add explicit type annotations to resolve TS7006/TS7031 errors:

    **Common patterns to fix:**
    - Callback parameters: `.map(e => ...)` → `.map((e: EntryType) => ...)`
    - Destructured bindings: `const { x } = obj` where obj is untyped
    - Prisma query result callbacks: type from Prisma generated types
    - Express error handlers: `(err: unknown)` or `(err: Error)`
    - Array method lambdas: `.filter(a => ...)`, `.reduce((acc, item) => ...)`

    **Strategy — work from smallest files to largest:**
    1. Single-error files first: pending-terms.ts (1), alerts.ts (1)
    2. Small files (2-3): admin.ts, call-logs.ts, cs-reps.ts, products.ts, ai-budget.ts, lead-timing.ts, convosoKpiPoller.ts
    3. Medium files (4-9): call-audits.ts, chargebacks.ts, change-requests.ts, auditQueue.ts, agentKpiAggregator.ts, payroll.ts
    4. Largest file last: sales.ts (56 errors)

    **For TS2339 errors** (property access on `{}`): Add proper interface/type for the object being accessed.
    **For TS2322 errors** (type assignment mismatch): Add explicit type assertions or fix the source type.

    **Rules:**
    - Use specific Prisma-generated types where available (import from @prisma/client)
    - Prefer `unknown` over `any` for truly unknown values
    - Do NOT change runtime behavior — annotation-only changes
    - Do NOT add `as any` casts — that defeats the purpose
    - If a type is genuinely complex, use a local type alias rather than inline
  </action>
