<objective>
## Goal
Create two one-time scripts: (1) clean up ~8 orphaned Clawback/PayrollEntry records left by the broken chargeback delete, restoring correct commission; (2) backfill the activity feed audit log with historical Sale, Chargeback, and PendingTerm entries.

## Purpose
Data integrity is the foundation for all v2.8 work. Orphaned records distort commission totals, and the empty activity feed reduces the command center's value. Both must be fixed before adding tests or caching.

## Output
- `prisma/scripts/cleanup-orphaned-clawbacks.ts` — orphan cleanup script
- `prisma/scripts/backfill-audit-log.ts` — audit log backfill script
- Both runnable via `npx tsx prisma/scripts/<script>.ts`
</objective>

## AC-1: Orphaned Clawback records are deleted
```gherkin
Given Clawback records exist whose saleId has no corresponding ChargebackSubmission.matchedSaleId
When the cleanup script runs
Then those Clawbacks and their ClawbackProducts are deleted
And the script logs the full record (id, saleId, agentId, amount, status, notes) before deletion
And the script logs each deleted Clawback (id, saleId, amount)
```

## AC-2: Orphaned PayrollEntries are deleted and commission recalculated
```gherkin
Given PayrollEntries with status ZEROED_OUT_IN_PERIOD or CLAWBACK_CROSS_PERIOD exist for sales with no remaining Clawback
When the cleanup script runs
Then those PayrollEntries are deleted
And upsertPayrollEntryForSale is called for each affected sale
And the script logs each affected sale (id, agent, commission restored)
And if any individual recalculation fails, the error is logged and the script continues with remaining sales
```

## AC-3: Activity feed is backfilled with historical records
```gherkin
Given Sale, ChargebackSubmission, and PendingTerm records exist with no corresponding AppAuditLog entry
When the backfill script runs
Then AppAuditLog entries are created with action "CREATE", correct entityType, entityId, and createdAt from the source record
And duplicate entries are not created on re-run (idempotent via entityType + entityId pair check)
```

## AC-4: Scripts are idempotent
```gherkin
Given the scripts have already been run once
When either script is run again
Then no duplicate records are created
And no errors are thrown
And output indicates "nothing to clean" or "already backfilled"
```

  <action>
    Create `prisma/scripts/cleanup-orphaned-clawbacks.ts`:

    1. Instantiate PrismaClient directly (standalone script — do NOT import from @ops/db or workspace packages, as workspace resolution won't work for standalone tsx scripts). Use:
       ```ts
       import { PrismaClient } from "@prisma/client";
       const prisma = new PrismaClient();
       ```
       <!-- audit-added: workspace import won't resolve in standalone tsx context -->

    2. Do NOT import upsertPayrollEntryForSale from the payroll service — it depends on @ops/db singleton. Instead, inline the minimal recalculation logic: delete orphaned entries, then let the existing payroll system pick up the sale on next period interaction. OR: use $executeRaw to recalculate. The simplest correct approach: just delete the orphaned clawback entries and zeroed payroll entries — the sale's premium is unaffected (as confirmed in the investigation), and the next upsert triggered by any payroll action on that sale will recreate the correct entry.
       <!-- audit-added: avoids fragile cross-workspace import -->

    3. Find orphaned Clawbacks:
       - Query all Clawback records with their sale's ChargebackSubmission count:
         ```ts
         const clawbacks = await prisma.clawback.findMany({
           include: { sale: { include: { _count: { select: { chargebackMatches: true } } } } }
         });
         ```
       - Actually, since there's no direct relation, use a subquery approach:
         - Get all distinct matchedSaleIds from ChargebackSubmission (where not null)
         - Get all Clawback records
         - Orphaned = clawback.saleId NOT IN the matchedSaleIds set

    4. For each orphaned Clawback, log the FULL record before deletion (audit trail):
       - Log: JSON.stringify({ id, saleId, agentId, amount, status, notes, createdAt })
       <!-- audit-added: log full record before destructive operation for post-incident reconstruction -->

    5. Process each orphaned Clawback within a transaction: <!-- audit-added -->
       - Delete related ClawbackProducts for this clawback
       - Delete the Clawback itself
       - Delete PayrollEntries for this sale with status IN ('ZEROED_OUT_IN_PERIOD', 'CLAWBACK_CROSS_PERIOD')
       - Wrap in prisma.$transaction to prevent partial state

    6. After all deletions, for each affected saleId, attempt commission recalculation:
       - Use raw SQL or a simple Prisma query to check if a PayrollEntry exists for the sale
       - If no entry exists, create one by querying the sale's commission data
       - Wrap each recalculation in try/catch — log errors but continue with remaining sales
       <!-- audit-added: per-record error resilience -->

    7. Print summary: X clawbacks deleted, Y entries deleted, Z sales affected

    8. If nothing found, print "No orphaned records found — database is clean"

    9. Call prisma.$disconnect() in a finally block
       <!-- audit-added: ensure disconnect even on error -->
  </action>

<task type="auto">
  <name>Task 2: Create activity feed backfill script</name>
  <files>prisma/scripts/backfill-audit-log.ts</files>
  <action>
    Create `prisma/scripts/backfill-audit-log.ts`:

    1. Instantiate PrismaClient directly (standalone script — same as Task 1, do NOT use @ops/db):
       ```ts
       import { PrismaClient } from "@prisma/client";
       const prisma = new PrismaClient();
       ```

    2. Fetch all existing AppAuditLog entries into a dedup Set keyed on (entityType, entityId) pair:
       <!-- audit-added: entityType+entityId pair prevents cross-type collisions -->
       - `SELECT entity_type, entity_id FROM app_audit_log WHERE action = 'CREATE'`
       - Store as `existingKeys: Set<string>` using key format `${entityType}:${entityId}`

    3. Backfill Sales:
       - Query all Sale records with select: id, createdAt, agentId, memberName, premium, status
       - For each sale where `Sale:${sale.id}` NOT in existingKeys:
         - Create AppAuditLog with:
           - actorUserId: null (system backfill)
           - action: "CREATE"
           - entityType: "Sale"
           - entityId: sale.id
           - metadata: { memberName, premium: Number(premium), status, backfilled: true }
           - createdAt: sale.createdAt
       - Use individual prisma.appAuditLog.create() calls (not createMany) to set explicit createdAt

    4. Backfill ChargebackSubmissions:
       - Query all ChargebackSubmission records with select: id, batchId, submittedAt, memberCompany, chargebackAmount
       - For each where `ChargebackSubmission:${id}` NOT in existingKeys:
         - Create AppAuditLog with:
           - actorUserId: null
           - action: "CREATE"
           - entityType: "ChargebackSubmission"
           - entityId: submission.id
           - metadata: { batchId, memberCompany, chargebackAmount: Number(chargebackAmount), backfilled: true }
           - createdAt: submission.submittedAt

    5. Backfill PendingTerms:
       - Query all PendingTerm records with select: id, batchId, createdAt, memberName, holdReason
       - For each where `PendingTerm:${id}` NOT in existingKeys:
         - Create AppAuditLog with:
           - actorUserId: null
           - action: "CREATE"
           - entityType: "PendingTerm"
           - entityId: term.id
           - metadata: { batchId, memberName, holdReason, backfilled: true }
           - createdAt: term.createdAt

    6. Wrap each entity type's backfill in try/catch — log errors for individual records but continue
       <!-- audit-added: per-record error resilience -->

    7. Print summary: X sales, Y chargebacks, Z pending terms backfilled

    8. If all already exist, print "Activity feed already backfilled — nothing to do"

    9. Call prisma.$disconnect() in a finally block
       <!-- audit-added: ensure disconnect even on error -->

    Note: Use `metadata: { backfilled: true }` flag so backfilled entries are distinguishable from real-time entries.
  </action>
