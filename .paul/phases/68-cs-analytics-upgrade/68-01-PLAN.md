<objective>
## Goal
Upgrade the CS Analytics tab with outreach-driven accountability metrics: two per-type leaderboards (chargebacks + pending terms) attributed via `assignedTo`, a save-rate-by-attempt-count correlation chart, and a gate-bypass callout. Folds into existing `CSAnalytics.tsx`. Owner/SUPER_ADMIN only. Date range picker driven.

## Purpose
Phases 65-67 instrumented CS outreach effort (ContactAttempt, SAVED/CANCELLED/NO_CONTACT outcomes, bypassReason). Phase 68 turns that data into accountability insight: who's attempting to save sales, who's actually saving them, and at what rate. Validates whether the 3-call gate produces real save-rate lift or is theater. Pays off the v2.9 instrumentation bet with decision-ready metrics for owner performance review.

## Output
- Extended aggregator service with per-type leaderboard queries + attempt-bucket correlation + bypass rollup
- One new API endpoint (`/cs/analytics/outreach`) returning the full payload
- Upgraded `CSAnalytics.tsx` rendering two sortable leaderboards, Recharts correlation chart, bypass callout, per-type CSV export
</objective>

## AC-1: Per-type outreach leaderboard metrics
```gherkin
Given chargebacks and pending terms exist with `assignedTo` populated to CsRepRoster names
And each record may have zero or more ContactAttempts logged by that rep
And records may have resolutionType of SAVED, CANCELLED, NO_CONTACT, or be still open
When an owner requests GET /cs/analytics/outreach?from=ISO&to=ISO
Then the response includes two leaderboards (chargebacks, pendingTerms)
And each row represents one CS rep with columns: assigned, worked, saved, cancelled, open, saveRate, workedRate, avgAttempts, avgTimeToResolveHours
And "assigned" counts records where assignedTo = rep within the date range
And "worked" counts assigned records where the rep has ≥1 ContactAttempt on that record
And "saveRate" = saved / (saved + cancelled), returning 0 when denominator is 0
And records with `submittedAt` prior to 2026-04-13 (v2.9 cutoff) are excluded from attempt-based metrics
And endpoint is role-gated to OWNER_VIEW and SUPER_ADMIN (401/403 otherwise)
```

```gherkin

And the error is logged via logError with a stable event name (e.g., "cs_outreach_analytics.subquery_failed")

```

## AC-9: Attribution + roster reconciliation behavior
<!-- audit-added -->
```gherkin
Given records have `assignedTo` as free-text strings populated by round-robin assignment
When the aggregator builds rep rows
Then `assignedTo` is trimmed and matched case-insensitively against `CsRepRoster.name` for grouping

And SAVED/CANCELLED credit is always given to the assignee (NOT the resolver) — this is the documented attribution model and must not be changed without stakeholder sign-off
```

