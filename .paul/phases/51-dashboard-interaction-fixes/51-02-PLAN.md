- Debounced search inputs on CS tracking and payroll agent sidebar

<task type="auto">
  <name>Task 1: Replace silent catches with toast feedback</name>
  <files>All 20 files listed in files_modified</files>
  <action>
    **Mechanical replacement pattern across all files:**

    For files that ALREADY have `useToast` imported (7 files):
    - Replace `.catch(() => ({}))` with `.catch(() => { toast("error", "Request failed"); })`
    - Replace empty `catch { }` or `catch { /* ignore */ }` with `catch { toast("error", "Request failed"); }`
    - Where the catch is on a response parse (e.g., `.json().catch(() => ({}))`), KEEP IT — these are safe fallbacks for JSON parsing, not error swallowing

    For files that DON'T have `useToast` (13 files):
    - Add `import { useToast } from "@ops/ui";` (or add to existing @ops/ui import)
    - Add `const { toast } = useToast();` at component top
    - If the component isn't inside a `<ToastProvider>`, check if its parent page already wraps it. The layout.tsx already has ToastProvider — so most components are covered. Only add ToastProvider if the component renders standalone.
    - Apply same catch replacement pattern

    **IMPORTANT distinctions:**
    - `.json().catch(() => ({}))` after a response → KEEP (JSON parse fallback, not error swallowing)
    - `.catch(() => {})` on a fetch/authFetch call → REPLACE (error swallowing)
    - `catch { }` in try/catch blocks → REPLACE
    - `catch { /* ignore */ }` → REPLACE
    - `.then(...).catch(() => {})` at end of chain → REPLACE

    **Toast message format:**
    - Use descriptive messages based on context: "Failed to load agents", "Failed to save config", "Failed to archive data"
    - Don't use generic "Request failed" when the context makes a better message obvious

    Avoid: Changing any logic inside try blocks or success paths.
    Avoid: Adding toast to .json().catch() fallbacks — these are parse safety, not error handling.
    Avoid: Modifying the login page (apps/ops-dashboard/app/page.tsx) — auth errors have their own handling.
  </action>
  <verify>

    - `grep -rn "catch {.*}" apps/ops-dashboard/app/\(dashboard\)/` returns zero empty catches
    - TypeScript compiles without new errors

<task type="auto">
  <name>Task 2: Add debounce to search inputs</name>
  <files>apps/ops-dashboard/app/(dashboard)/cs/CSTracking.tsx, apps/ops-dashboard/app/(dashboard)/payroll/AgentSidebar.tsx</files>
  <action>
    **Create a simple useDebounce hook inline (no new file — keep it local):**

    In each file, add:
    ```
    function useDebounce<T>(value: T, delay: number): T {
      const [debounced, setDebounced] = useState(value);
      useEffect(() => {
        const timer = setTimeout(() => setDebounced(value), delay);
        return () => clearTimeout(timer);
      }, [value, delay]);
      return debounced;
    }
    ```

    **CSTracking.tsx:**
    - Keep `searchTerm` as the instant state (user types, input updates immediately)
    - Add `const debouncedSearch = useDebounce(searchTerm, 300);`
    - Replace all filtering logic that uses `searchTerm` with `debouncedSearch`
    - The input's onChange still sets `searchTerm` (instant visual feedback)

    **AgentSidebar.tsx:**
    - Same pattern: keep `searchQuery` for input, add `debouncedQuery = useDebounce(searchQuery, 300)`
    - Replace filtering logic to use `debouncedQuery`

    Avoid: Creating a shared hooks file — inline in each component for now.
    Avoid: Debouncing date range inputs — they're date pickers, not search fields.
  </action>
  <verify>
    - Type rapidly in CS tracking search → list doesn't filter until 300ms pause
    - Type rapidly in payroll agent sidebar search → same debounce behavior
    - Input still shows characters immediately (no visual lag)
  </verify>
