<objective>
## Goal
Fix the payroll sidebar to show commission for the current week's payroll period instead of the most recent period by date — preventing ACH-deferred entries from skewing the at-a-glance totals.

## Purpose
When ACH sales create entries in next week's period (due to the extra week delay), `currentPeriodId` picks that future period as "most recent" since it has the latest `weekStart`. The sidebar then shows only the deferred agent's commission, misleading managers about this Friday's actual payroll.

## Output
- `currentPeriodId` logic fixed to select the period containing today's date
- Sidebar shows accurate commission for the current payroll period
</objective>

    With date-aware logic that finds the period containing today:

    ```typescript
    const currentPeriodId = useMemo(() => {
      if (periods.length === 0) return null;
      const today = new Date().toISOString().slice(0, 10);
      // Find period where today falls between weekStart and weekEnd
      const current = periods.find(p =>
        p.weekStart.slice(0, 10) <= today && today <= p.weekEnd.slice(0, 10)
      );
      if (current) return current.id;
      // Fallback: most recent period that has already started (not future)
      const past = periods
        .filter(p => p.weekStart.slice(0, 10) <= today)
        .sort((a, b) => new Date(b.weekStart).getTime() - new Date(a.weekStart).getTime());
      return past[0]?.id ?? periods[0]?.id ?? null;
    }, [periods]);
    ```

    This ensures:
    1. If today is within a period's date range → use that period
    2. If today doesn't fall in any period → use the most recent past period
    3. If no past periods exist → fall back to first available period

    Avoid: Using Luxon here — simple ISO string comparison is sufficient for date-only checks.
    Avoid: Changing how agentData or sortedAgents consume currentPeriodId — only the selection logic changes.
