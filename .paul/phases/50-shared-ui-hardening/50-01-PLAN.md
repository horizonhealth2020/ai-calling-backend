- Enhanced Button, Input, Select with disabled/ARIA states
- New ConfirmModal component exported from @ops/ui
- All changes in `packages/ui/` only (no dashboard changes)

<acceptance_criteria>

## AC-1: Disabled Button is visually distinct from enabled
```gherkin
Given a Button with disabled={true}
When rendered on screen
Then it shows a desaturated background, reduced text contrast, and no cursor pointer
And it is visually distinguishable from an enabled button at a glance (not just opacity)
And it has aria-disabled="true" in the DOM
```

## AC-2: Disabled Input and Select are visually distinct
```gherkin
Given an Input or Select with disabled={true}
When rendered on screen
Then it shows a darker inset background and muted text color
And the border is more subtle than the enabled state
And it has aria-disabled="true" in the DOM
```

## AC-3: Error inputs have aria-invalid
```gherkin
Given an Input or Select with error prop set
When rendered on screen
Then it has aria-invalid="true" in the DOM
And the error message is linked via aria-describedby
```

## AC-4: Loading Button has aria-busy
```gherkin
Given a Button with loading={true}
When rendered on screen
Then it has aria-busy="true" in the DOM
And the spinner has aria-hidden="true" to prevent screen reader noise
```

## AC-5: ConfirmModal renders with dark glassmorphism theme
```gherkin
Given ConfirmModal is rendered with title, message, and onConfirm/onCancel
When the modal opens
Then it shows a backdrop overlay with a centered glassmorphism card
And the card has title, message body, and Cancel/Confirm buttons
And the Confirm button matches the variant prop (danger for destructive, primary for neutral)
And pressing Escape or clicking backdrop calls onCancel
And focus is trapped inside the modal while open
```

## AC-6: ConfirmModal is accessible
```gherkin
Given ConfirmModal is open
When a screen reader encounters it
Then it has role="dialog" and aria-modal="true"
And aria-labelledby points to the title
And aria-describedby points to the message
And the Confirm button receives initial focus
```

<task type="auto">
  <name>Task 2: Create ConfirmModal component</name>
  <files>packages/ui/src/components/ConfirmModal.tsx, packages/ui/src/components/index.ts</files>
  <action>
    Create a new ConfirmModal component following the dark glassmorphism patterns from Toast.tsx:

    **Props interface:**
    ```
    interface ConfirmModalProps {
      open: boolean;
      title: string;
      message: string;
      confirmLabel?: string;    // default: "Confirm"
      cancelLabel?: string;     // default: "Cancel"
      variant?: "primary" | "danger";  // default: "primary"
      loading?: boolean;        // for async confirm handlers
      onConfirm: () => void;
      onCancel: () => void;
    }
    ```

    **Structure:**
    - Backdrop: fixed fullscreen, `background: rgba(0,0,0,0.6)`, `backdropFilter: blur(4px)`, z-index 10000, flex center
    - Card: `colors.bgSurfaceOverlay` background, `colors.borderDefault` border, `radius.xl` border-radius, `shadows.xl` box-shadow, max-width 420px, padding 24px
    - Title: `typography.sizes.md`, `colors.textPrimary`, fontWeight 600
    - Message: `typography.sizes.sm`, `colors.textSecondary`, margin-top 8px
    - Button row: flex, justify-end, gap 10, margin-top 24px
    - Cancel button: Button variant="secondary" size="sm"
    - Confirm button: Button variant={variant} size="sm" loading={loading}

    **Behavior:**
    - Render nothing when `open` is false (early return null)
    - Escape key calls onCancel (useEffect with keydown listener)
    - Clicking backdrop calls onCancel (onClick on backdrop, stopPropagation on card)
    - Focus trap (WCAG 2.4.3 compliant): <!-- audit-added -->
      - On open: focus the confirm button via useRef + useEffect
      - Intercept Tab keydown: query all focusable elements inside the dialog (buttons), cycle focus from last→first on Tab and first→last on Shift+Tab
      - On close: no restore needed (modal unmounts, browser handles it)
    - Generate unique IDs per instance using a module-level counter (`let idCounter = 0; const id = useRef(++idCounter).current`) to avoid duplicate IDs if multiple modals coexist <!-- audit-added -->
    - Add `role="dialog"`, `aria-modal="true"`, `aria-labelledby={titleId}`, `aria-describedby={messageId}` to the card (where titleId/messageId use the counter-based ID)
    - Add corresponding ids to title and message elements

    **Export from index.ts:**
    - Add `export { ConfirmModal } from "./ConfirmModal";` to index.ts
    - Also export the type: `export type { ConfirmModalProps } from "./ConfirmModal";`

    Avoid: Creating a portal — inline rendering is fine for this use case since z-index handles stacking.
    Avoid: Animation on open/close — keep it simple for now (Phase 52 will add transitions).
    Avoid: useId() hook — use module-level counter pattern instead (`let idCounter = 0` at file scope, `const id = useRef(++idCounter).current` in component) for broader compatibility. <!-- audit-added -->
  </action>
  <verify>
    - TypeScript compiles without errors: `npx tsc --noEmit -p packages/ui/tsconfig.json` (or nearest tsconfig)
    - Import ConfirmModal from @ops/ui in a dashboard file to verify export works
    - Manually test: open modal, press Escape (should close), click backdrop (should close), tab between buttons (focus stays inside)
  </verify>
  <done>AC-5, AC-6 satisfied: ConfirmModal renders with themed styling, proper ARIA, focus management</done>
