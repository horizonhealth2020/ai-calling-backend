<objective>
## Goal
Fix UAT-003: Sale Edit + Save on a clawback-affected paycard row (status `ZEROED_OUT_IN_PERIOD` / `CLAWBACK_APPLIED` / `CLAWBACK_CROSS_PERIOD`) silently undoes the chargeback.

## Purpose
Pre-existing bug surfaced during Phase 81 UAT. Current behavior: `upsertPayrollEntryForSale` blindly overwrites `payoutAmount` + `netAmount` in its `update` clause, rewriting a ZEROED row back to full commission while leaving `status = ZEROED_OUT_IN_PERIOD`. Result: agent is paid commission that was supposed to be clawed back, AND the row is visually inconsistent (yellow tint claiming zeroed, but subtotal includes the full amount). This violates the project's commission-accuracy invariant.

## Output
- Server-side guard in `upsertPayrollEntryForSale`: when existing entry is clawback-affected, preserve zeroed `payoutAmount`/`netAmount`/`status` — only update `halvingReason` (metadata that doesn't affect agent pay).
- Unit test case asserting the guard fires for all 3 clawback statuses.

Priority: 1 Major (pre-existing bug, reopened via Phase 81 UAT scope)

    ```
    // Phase 81-02 (UAT-003): Preserve clawback-affected state. If an existing
    // entry is in a clawback status, refreshing payoutAmount/netAmount would
    // silently undo the chargeback (agent gets paid for a clawed-back sale).
    // The only safe path out of a clawback state is an explicit reversal:
    // - payroll approves a RECOVERY alert (reverseClawback deletes the row first, so upsert's CREATE branch runs)
    // - OR admin DELETE /chargebacks/:id (deletes Clawback + clawback entries, then recalcs)
    const clawbackStates = new Set(["ZEROED_OUT_IN_PERIOD", "CLAWBACK_APPLIED", "CLAWBACK_CROSS_PERIOD"]);
    if (existing && clawbackStates.has(existing.status)) {
      // Metadata-only update — preserves payout/net/status as-is.
      return db.payrollEntry.update({
        where: { payrollPeriodId_saleId: { payrollPeriodId: period.id, saleId } },
        data: { halvingReason },
      });
    }

    return db.payrollEntry.upsert({
      where: { payrollPeriodId_saleId: { payrollPeriodId: period.id, saleId } },
      create: {
        payrollPeriodId: period.id,
        saleId,
        agentId: sale.agentId,
        payoutAmount,
        netAmount: payoutAmount,
        halvingReason,
      },
      update: { payoutAmount, netAmount, halvingReason },
    });
    ```

    **Rationale:**
    - The upsert's existing CREATE branch stays unchanged — when no entry exists (e.g. after `reverseClawback` deletes the ZEROED row), a fresh live entry is created with status defaulting to PENDING (Prisma default).
    - The clawback-preservation guard ONLY fires when BOTH conditions hold: existing entry is present AND in a clawback status. Any path that legitimately wants to restore (recovery flow, admin chargeback delete) already DELETES the clawback entry first, so the guard's `existing` is null at that point.
    - No logAudit call is added inline (calling logAudit requires userId which upsertPayrollEntryForSale doesn't receive). Observability is via existing audit at the CALLER side (e.g. PATCH /sales/:id already logs "UPDATE Sale" with field deltas; a sibling entry skipped on preservation is implicit). Future Phase could add a logAudit call here if observability gaps surface.
