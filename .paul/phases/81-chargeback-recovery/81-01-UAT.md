### UAT-002: No RECOVERY alert on CS resolve after sale was deleted + re-entered

**Discovered:** 2026-04-17
**Phase/Plan:** 81-01
**Severity:** Minor (expected behavior given the fixture state)
**Feature:** AC-1
**User scenario:** "I deleted the sale entry, re-entered it, resolved chargeback — got no alert."
**Analysis:** `routes/sales.ts:680-688` — deleting a Sale ALSO `deleteMany`'s any Clawbacks for that saleId (pre-existing v1.x cleanup behavior). So after delete + re-enter: the original Clawback no longer exists. CS resolve "recovered" correctly finds no Clawback → takes AC-2 path (no recovery alert created; logs `chargeback_resolved_recovered_no_clawback`).
**Resolution:** Working as designed. This is correct behavior per AC-1/AC-2 spec: "no clawback → no recovery alert to approve." If user expected the recovery flow to work after sale re-entry, that would require:
  - Re-submitting the chargeback (creates new ChargebackSubmission with new id)
  - Payroll re-approves the new submission (creates new Clawback)
  - THEN CS can mark the new chargeback recovered → recovery alert works as expected

### UAT-003 [FIXED in 81-02-FIX — 2026-04-17]: Edit + Save on a ZEROED paycard row "removes chargeback" visually

**Discovered:** 2026-04-17 (surfaced during UAT of Phase 81, but not a Phase 81 regression)
**Phase/Plan:** Pre-existing, predates Phase 81
**Severity:** Cosmetic/Major (data inconsistency — user can visually "undo" a chargeback by editing the sale)
**Feature:** Sale edit on ZEROED_OUT_IN_PERIOD rows
**Description:** When a payroll row is in status=ZEROED_OUT_IN_PERIOD (chargeback applied same-week), pressing Edit on the row and then pressing Save (even with no changes) triggers `PATCH /sales/:id` which calls `upsertPayrollEntryForSale`. That upsert's `update` clause overwrites `payoutAmount` and `netAmount` with freshly-calculated commission values — but leaves `status=ZEROED_OUT_IN_PERIOD` intact. Result: the row shows non-zero commission with yellow ZEROED tint (visually inconsistent) AND the agent's net subtotal reflects the "restored" payoutAmount even though no Clawback reversal was approved.
**Relevant code:** `apps/ops-api/src/services/payroll.ts:421-432` `upsertPayrollEntryForSale` `update` clause does not preserve status or guard against overwriting a clawback-affected entry.
**Expected:** Editing a sale should NOT silently undo a chargeback. Either:
  - (a) Block Edit on rows with status in (ZEROED_OUT_IN_PERIOD, CLAWBACK_APPLIED, CLAWBACK_CROSS_PERIOD) with a message "Reverse the chargeback first via payroll Approve Recovery"
  - (b) Preserve the chargeback-affected state on upsert: if existing entry has clawback-ish status, keep payoutAmount=0 + status=ZEROED regardless of upsert's re-derived commission
  - (c) Auto-delete the associated Clawback on sale edit (creates its own audit concerns)
**Action:** User chose to fix within Phase 81 scope. Spun Plan 81-02-FIX.md. User note: "PRESSING EDIT AND BACKING OUT SHOULD NOT REMOVE A CHARGEBACK" — confirmed that Cancel (X) button does NO mutation; only Save triggers the upsert + undo.
**Resolution (2026-04-17):** Applied server-side guard in `upsertPayrollEntryForSale` (`apps/ops-api/src/services/payroll.ts:428-434`). When existing PayrollEntry has status in (`ZEROED_OUT_IN_PERIOD`, `CLAWBACK_APPLIED`, `CLAWBACK_CROSS_PERIOD`), upsert takes a metadata-only update branch (sets `halvingReason` only) instead of overwriting `payoutAmount`/`netAmount`/`status`. Phase 81's `reverseClawback` flow unaffected because it DELETES the clawback row before calling upsert, making `existing` null and the normal CREATE branch run. 222/222 tests pass; no regression. New unit tests for the guard were scope-narrowed out (same rationale as Phase 79 DEFER-01: upsertPayrollEntryForSale has module-internal lexical refs to resolveBundleRequirement/calculateCommission that resist jest mocking; relying on structural grep + full-suite regression + manual UAT as compensating controls). Manual UAT re-verification pending.
