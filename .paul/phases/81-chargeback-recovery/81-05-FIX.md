<objective>
## Goal
Fix 500 on POST /alerts/:id/approve for RECOVERY-type alerts when `Clawback.appliedPayrollPeriodId` doesn't match the actual ZEROED/CROSS_PERIOD entry's period.

## Purpose
Live UAT on 2026-04-17 surfaced 500 on Approve Recovery. DB diagnosis revealed:
- approveAlert (SUBMISSION path) sets `Clawback.appliedPayrollPeriodId = findOldestOpenPeriodForAgent(agentId)` BEFORE knowing whether applyChargebackToEntry will take in_period or cross_period mode
- For in_period mode (original entry's period is OPEN), applyChargebackToEntry ZEROs the original entry (which is in its own sale's period — may be a different OPEN period than "oldest open")
- Result: Clawback record's `appliedPayrollPeriodId` can diverge from the actual ZEROED entry's `payrollPeriodId`
- reverseClawback required exact match → failed to locate entry → threw → 500

Source: Live UAT (Donna Zarembski). Clawback.appliedPayrollPeriodId = `2026-04-05...2026-04-11`, actual ZEROED entry was in `2026-04-12...2026-04-18`. Mismatch.

## Output
reverseClawback locates the affected entry by STATUS alone (ZEROED_OUT_IN_PERIOD / CLAWBACK_CROSS_PERIOD) — scoped to the sale's entries, which are unique per (period, sale) via @@unique constraint. Verifies the ENTRY'S actual period is OPEN (not the Clawback's recorded period). Updates periodId returned to be the entry's real period.
</objective>
