<objective>
## Goal
Enable CS-driven chargeback recovery. When a CS rep resolves a chargeback with `resolutionType === "recovered"` AND a `Clawback` already exists for that chargeback, create a `PayrollAlert` with `type="RECOVERY"` pointing to that clawback. Payroll sees an inline "RECOVERED" banner on the affected agent's paycard, approves, and the Clawback is reversed: same-week `ZEROED_OUT_IN_PERIOD` entries restore via `upsertPayrollEntryForSale` (server single-source-of-truth); cross-period `CLAWBACK_CROSS_PERIOD` entries are deleted after state-verification; the Clawback row is deleted. OPEN-period only (400 on LOCKED/FINALIZED). Forward-only. Mirrors Phase 79 approval-gate pattern in reverse.

## Purpose
Today when CS recovers a chargeback, the agent's payroll already reflects the deduction and has no path back to paid. This plan closes that loop: agents get paid what they earned once the money is actually recovered, without payroll manually unwinding Clawbacks. Payroll stays in control via the same alert/approve pattern used for submissions.

## Output
- One additive Prisma migration (nullable `type` with default + nullable `clawbackId` FK on `payroll_alerts`)
- `reverseClawback(tx, clawbackId)` helper in `services/payroll.ts` — uses `upsertPayrollEntryForSale` for in-period restore (audit-added M-1)
- `createRecoveryAlert(tx, ...)` in `services/alerts.ts` — tx-aware, idempotent (audit-added M-2)
- Branch inside `approveAlert` for `type === "RECOVERY"` → reversal path with race-safe idempotency
- Hook in `PATCH /chargebacks/:id/resolve` on `resolutionType === "recovered"` — tx-wrapped; auto-clears orphan PENDING SUBMISSION alerts (audit-added M-3)
- New socket event `emitClawbackReversed` with enriched payload (audit-added SR-3)
- Inline recovery-alert banner on payroll AgentCard/WeekSection
- Unit tests for reversal (in-period + cross-period + LOCKED + FINALIZED + race-condition cases)
</objective>
