<objective>
## Goal
Fix two production bugs: (1) sales receipt parser silently misreads payment type as "CC" when the `Type:` line is blank, (2) mid-week fronted advances are double-paid by being added to current-week net.

## Purpose
Both bugs cause financial inaccuracy — parser misroutes ACH sales to CC (wrong payroll week via `shiftWeeks`), and fronted-additive inflates current paycards for money already handed to the agent. Direct hits on the project's core value ("agents get paid right").

## Output
- Parser correctly returns `paymentType: "ACH"` for BANK/routing-number receipts, even when `Type:` line is blank.
- Net formula removes `+ fronted` — fronted recovery continues via existing carryover-to-hold on lock.
- One new unit test locking in the net formula.
- Decision reversal recorded: PROJECT.md `Fronted additive in net formula` → `Fronted is advance, not a net addition`.
</objective>

  <action>
    In the parser block around line 202:

    1. Replace the two-regex chain with a line-scoped detection that will NOT cross newlines:
       - Split text into lines, find the line starting with `Type:` inside the Payment section (prefer the line AFTER a line equal to "Payment" or containing "Payment" header so we don't catch the "Date: ... Type: SALE" header).
       - On that Type line only, test for `\b(BANK|ACH|CARD|CC|CREDIT)\b` (case-insensitive, single-line match).

    2. If the Type line is blank OR its value isn't a known payment keyword, fall back to inspecting the `Method:` line in the Payment section:
       - If it contains a 9-digit routing-number token (regex: `\b\d{9}\b`), OR the word "Bank" (case-insensitive, standalone word — careful of "Bankruptcy" style — use `\bBank\b`), OR "Checking"/"Savings" — set paymentType = "ACH".
       - Otherwise leave paymentType undefined (user picks manually).

    3. Keep existing mapping: BANK/ACH → "ACH"; CARD/CC/CREDIT → "CC".

    Avoid:
      - Regex crossing newlines (the root cause of the bug — `\s*(\w+)` swallowed the newline and captured "Method").
      - Matching "Type: SALE" from the Date/Order header line — scope detection to the Payment section (after a line containing "Payment" as a header, or by requiring the Type line to be near a "Method:" line).
      - False positives from routing-number-like digits in addresses — only inspect the Method line.
