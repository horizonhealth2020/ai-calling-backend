<objective>
Fix Sub-feature 5: closed-period chargebacks currently fall back and silently mutate the locked/finalized PayrollEntry. Instead, detect when the sale's only PayrollEntry lives in a LOCKED or FINALIZED period and insert a NEW negative PayrollEntry in the oldest OPEN period (agent-agnostic) with status `CLAWBACK_CROSS_PERIOD` (orange). In-period chargebacks continue to zero in place but now use status `ZEROED_OUT_IN_PERIOD` (yellow) for visual distinction.

**Critical drift corrections honored:**
- Enum values are `OPEN / LOCKED / FINALIZED` (NOT `OPEN / CLOSED / PAID` as CONTEXT.md says)
- TWO code paths share the bug: `routes/payroll.ts:187-288` POST /clawbacks AND `routes/chargebacks.ts:194-265` batch path — both need the fix via the shared helper
- `approveAlert` path in `services/alerts.ts` currently does NOT mutate PayrollEntry — extending it for parity is recommended
- A new `findOldestOpenPeriod()` helper (agent-agnostic) is needed because the existing `findOldestOpenPeriodForAgent` filters on `entries.some.agentId` which returns null for fresh periods
- `PayrollEntry.@@unique([payrollPeriodId, saleId])` makes the cross-period insert safe — same sale can have entries in multiple periods

Purpose: Agents should see the deduction on a current open period (visible in dashboard + print) rather than have their locked payroll be silently back-edited.
Output: New schema enum values + shared helper + three code-path updates + frontend row highlighting + print card parity.
</objective>

**Step B — Add `applyChargebackToEntry()` shared helper:**

```ts
export async function applyChargebackToEntry(
  tx: Prisma.TransactionClient,
  sale: {
    id: string;
    agentId: string;
    payrollEntries: { id: string; payrollPeriodId: string; payoutAmount: any }[];
  },
  chargebackAmount: number,
): Promise<{ mode: "in_period" | "cross_period"; entryId: string }> {
  const originalEntry = sale.payrollEntries[0];
  const originalPeriod = originalEntry
    ? await tx.payrollPeriod.findUnique({
        where: { id: originalEntry.payrollPeriodId },
        select: { status: true },
      })
    : null;

  // IN-PERIOD (OPEN) — zero original row in place, mark yellow
  if (originalPeriod?.status === "OPEN" && originalEntry) {
    const updated = await tx.payrollEntry.update({
      where: { id: originalEntry.id },
      data: {
        payoutAmount: 0,
        netAmount: 0,
        status: "ZEROED_OUT_IN_PERIOD",
      },
    });
    return { mode: "in_period", entryId: updated.id };
  }

  // CROSS-PERIOD (LOCKED or FINALIZED) — insert new negative row in oldest OPEN period
  const oldestOpen = await tx.payrollPeriod.findFirst({
    where: { status: "OPEN" },
    orderBy: { weekStart: "asc" },
    select: { id: true },
  });
  if (!oldestOpen) {
    throw new Error("No OPEN payroll period exists for cross-period chargeback");
  }

  // Safe because @@unique([payrollPeriodId, saleId]) allows same sale in different periods
  const created = await tx.payrollEntry.create({
    data: {
      payrollPeriodId: oldestOpen.id,
      saleId: sale.id,
      agentId: sale.agentId,
      payoutAmount: 0,
      adjustmentAmount: -chargebackAmount,
      netAmount: -chargebackAmount,
      status: "CLAWBACK_CROSS_PERIOD",
    },
  });
  return { mode: "cross_period", entryId: created.id };
}
```
