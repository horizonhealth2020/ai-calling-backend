The existing `resolveBundleRequirement` function at line 218 has the ACA auto-fulfill check at lines 231-238:

```typescript
if (saleId) {
  const acaCovering = await prisma.sale.findFirst({
    where: { acaCoveringSaleId: saleId, product: { type: "ACA_PL" }, status: "RAN" },
  });
  if (acaCovering) {
    return { requiredAddonAvailable: true, fallbackAddonAvailable: false, halvingReason: null };
  }
}
```

Per decision D-03: "Any ACA PL product sold alongside a core sale automatically satisfies that core product's bundle requirement — regardless of the core product's requiredBundleAddonId configuration."

The current code has a critical sequencing issue: line 228 does `if (!coreProduct.requiredBundleAddonId) return null;` BEFORE the ACA check at line 231. When a core product has NO `requiredBundleAddonId` configured, the function returns null (meaning "no bundle requirement") and never checks for ACA. This is actually correct behavior — if there is no bundle requirement, there is nothing to halve, so commission is already full.

However, there is a subtle issue: the ACA check happens AFTER the early return for no-bundle-requirement, which means it only fires when a bundle requirement IS configured. Per D-03, this is the correct behavior — ACA satisfies the configured bundle requirement.

VERIFY the following path works correctly:
1. Core product HAS `requiredBundleAddonId` configured
2. The required addon is NOT in the sale's addons
3. But the sale HAS a linked ACA covering sale (via `acaCoveringSaleId`)
4. Expected result: `{ requiredAddonAvailable: true, fallbackAddonAvailable: false, halvingReason: null }` — full commission, no halving

The code at lines 231-238 handles this correctly. BUT verify that the `saleId` parameter is actually passed. Look at line 326: `resolveBundleRequirement(sale.product, sale.memberState, addonProductIds, saleId)` — yes, `saleId` is passed as the 4th argument.

POTENTIAL BUG TO CHECK: The `acaCoveringSaleId` field on the Sale model uses a self-relation where the ACA sale points TO the parent sale. The query at line 232-233 searches for `acaCoveringSaleId: saleId` — this finds ACA sales whose `acaCoveringSaleId` equals the current sale's ID. This means the ACA sale's `acaCoveringSaleId` field must be set to the PARENT core sale's ID. Verify this matches how the manager entry form creates ACA covering sales.

Read the manager entry form to confirm the ACA covering sale creation sets `acaCoveringSaleId` correctly:
