# Phase 5: Commission Preview & Sale Editing - Research

**Researched:** 2026-03-15
**Domain:** Commission calculation preview, sale editing with approval workflow, inline editing UX
**Confidence:** HIGH

## Summary

Phase 5 adds two distinct capabilities to the ops platform: (1) a live commission preview on the sales entry form that calls a dedicated server endpoint to show managers what commission a sale will generate before submission, and (2) full sale editing with an approval workflow where managers propose edits and payroll reviews/approves them. Both features build on well-established patterns already in the codebase -- the `calculateCommission()` function in `payroll.ts` for preview, and the `StatusChangeRequest` model pattern from Phase 10 for the approval workflow.

The codebase is a monorepo with Express.js API (`apps/ops-api`) and Next.js 15 frontends. All UI uses inline `React.CSSProperties` with a dark glassmorphism theme. The existing sales form is in `apps/manager-dashboard/app/page.tsx` (2025 lines) and already has a two-column layout (form left, receipt/addons right). The agent sales tab in the same file renders sales grouped by agent in table rows with inline status dropdowns -- this is where inline editing will be added.

**Primary recommendation:** Create a `POST /api/sales/preview` endpoint that calls `calculateCommission()` with a mock sale object, add a `SaleEditRequest` model parallel to `StatusChangeRequest`, and extend the existing agent sales tab with an edit button that expands inline fields.

### Pattern 1: Commission Preview Endpoint
**What:** A stateless POST endpoint that accepts a sale payload, builds a mock sale object, calls `calculateCommission()`, and returns the breakdown without touching the database.
**When to use:** Every time financial fields change on the sales entry form.
**Example:**
```typescript
// POST /api/sales/preview
router.post("/sales/preview", requireAuth, requireRole("MANAGER", "SUPER_ADMIN"), asyncHandler(async (req, res) => {
  const schema = z.object({
    productId: z.string(),
    premium: z.number().min(0),
    enrollmentFee: z.number().min(0).nullable().optional(),
    addonProductIds: z.array(z.string()).default([]),
    addonPremiums: z.record(z.string(), z.number().min(0)).default({}),
    paymentType: z.enum(["CC", "ACH"]),
    status: z.enum(["RAN", "DECLINED", "DEAD"]).optional().default("RAN"),
    commissionApproved: z.boolean().optional().default(false),
    saleDate: z.string().optional(),
  });
  const parsed = schema.safeParse(req.body);
  if (!parsed.success) return res.status(400).json(zodErr(parsed.error));

  // Fetch product data needed for calculation
  const product = await prisma.product.findUnique({ where: { id: parsed.data.productId } });
  if (!product) return res.status(400).json({ error: "Product not found" });

  const addonProducts = await prisma.product.findMany({
    where: { id: { in: parsed.data.addonProductIds } },
  });

  // Build mock sale object matching SaleWithProduct type
  const mockSale = {
    premium: parsed.data.premium,
    enrollmentFee: parsed.data.enrollmentFee ?? null,
    commissionApproved: parsed.data.commissionApproved,
    status: parsed.data.status,
    product,
    addons: addonProducts.map(p => ({
      product: p,
      premium: parsed.data.addonPremiums[p.id] ?? 0,
    })),
  } as any; // Cast to SaleWithProduct shape

  const commission = calculateCommission(mockSale);

  // Calculate period assignment for display
  const shiftWeeks = parsed.data.paymentType === 'ACH' ? 1 : 0;
  const saleDate = parsed.data.saleDate ? new Date(parsed.data.saleDate + "T12:00:00") : new Date();
  const { weekStart, weekEnd } = getSundayWeekRange(saleDate, shiftWeeks);

  res.json({
    commission,
    periodStart: weekStart.toISOString(),
    periodEnd: weekEnd.toISOString(),
    // Breakdown details for display
    breakdown: {
      hasBundleQualifier: addonProducts.some(p => p.isBundleQualifier),
      hasCore: product.type === "CORE",
      enrollmentFee: parsed.data.enrollmentFee,
      paymentType: parsed.data.paymentType,
    },
  });
}));
```

### Pattern 2: SaleEditRequest Model (Approval Workflow)
**What:** A new Prisma model that stores proposed sale edits as a JSON diff. Follows the same PENDING/APPROVED/REJECTED status pattern as StatusChangeRequest.
**When to use:** When a MANAGER edits a sale (not PAYROLL/SUPER_ADMIN who can edit directly).
**Example:**
```prisma
model SaleEditRequest {
  id           String              @id @default(cuid())
  saleId       String              @map("sale_id")
  requestedBy  String              @map("requested_by")
  requestedAt  DateTime            @default(now()) @map("requested_at")
  changes      Json                // { field: { old: value, new: value } }
  status       ChangeRequestStatus @default(PENDING)
  reviewedBy   String?             @map("reviewed_by")
  reviewedAt   DateTime?           @map("reviewed_at")
  createdAt    DateTime            @default(now()) @map("created_at")

  sale      Sale  @relation(fields: [saleId], references: [id], onDelete: Cascade)
  requester User  @relation("SaleEditRequester", fields: [requestedBy], references: [id])
  reviewer  User? @relation("SaleEditReviewer", fields: [reviewedBy], references: [id])

  @@index([saleId])
  @@index([status])
  @@map("sale_edit_requests")
}
```

### Pattern 3: Debounced Preview on Client
**What:** Use a `useRef` + `setTimeout` pattern (no external library needed) to debounce preview API calls when numeric fields change.
**When to use:** On premium and enrollment fee input changes.
**Example:**
```typescript
const previewTimer = useRef<ReturnType<typeof setTimeout>>();
const [previewData, setPreviewData] = useState<{ commission: number; periodStart: string; periodEnd: string } | null>(null);
const [previewLoading, setPreviewLoading] = useState(false);

function triggerPreview() {
  clearTimeout(previewTimer.current);
  previewTimer.current = setTimeout(async () => {
    if (!form.productId || !form.premium) return;
    setPreviewLoading(true);
    try {
      const res = await authFetch(`${API}/api/sales/preview`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          productId: form.productId,
          premium: Number(form.premium),
          enrollmentFee: form.enrollmentFee ? Number(form.enrollmentFee) : null,
          addonProductIds: form.addonProductIds,
          addonPremiums: /* ... */,
          paymentType: form.paymentType,
          saleDate: form.saleDate,
        }),
      });
      if (res.ok) setPreviewData(await res.json());
    } finally {
      setPreviewLoading(false);
    }
  }, 500);
}
```

### Anti-Patterns to Avoid
- **Client-side commission calculation:** Never calculate commission in the browser. Always use the server endpoint. The preview endpoint must call `calculateCommission()` directly.
- **Modifying `calculateCommission()` for preview:** The function should remain pure. Build a mock sale object that matches its expected input shape.
- **Editing sale directly for managers:** MANAGER edits must create a SaleEditRequest. Only PAYROLL/SUPER_ADMIN bypass the approval workflow.
- **Modifying finalized period entries:** When a sale in a finalized period is edited, do NOT modify the finalized entry. Create an adjustment in the next OPEN period (follows `handleCommissionZeroing` CLAWBACK_APPLIED pattern).
- **Fetching full sale details in the preview endpoint:** Only fetch product data needed for commission calculation. Do not create/update any database records.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Commission calculation | Custom preview-specific calculation | Existing `calculateCommission()` from `payroll.ts` | Guarantees preview matches actual -- the entire point of server-authoritative preview |
| Period assignment | Custom date logic | Existing `getSundayWeekRange()` from `payroll.ts` | ACH shift, timezone handling already solved |
| Approval workflow pattern | New approval abstraction | Follow `StatusChangeRequest` pattern (PENDING/APPROVED/REJECTED) | Proven pattern from Phase 10, same UI in payroll dashboard |
| Debounce | Custom debounce utility | `useRef` + `setTimeout` (built-in) | Simple enough for this use case, no library needed |
| JSON diff generation | Custom object diff | Simple `Object.keys` comparison on flat sale fields | Sale objects are flat (no deep nesting), diff is straightforward |

**Key insight:** The preview endpoint is trivial because `calculateCommission()` is already a pure function that takes a sale-shaped object. The hard part is building the correct mock object with fetched product data.

## Common Pitfalls

### Pitfall 1: Preview Payload Mismatch
**What goes wrong:** The preview endpoint accepts different field names or types than the create endpoint, causing preview to show wrong values.
**Why it happens:** Two separate Zod schemas drift apart over time.
**How to avoid:** Use the same Zod schema base for both endpoints. The preview schema should be a subset of the create schema (only financial fields + product selection).
**Warning signs:** Preview commission differs from actual commission after submission.

### Pitfall 2: Race Conditions in Debounced Preview
**What goes wrong:** User types quickly, multiple preview requests fire, and an older response arrives after a newer one, showing stale data.
**Why it happens:** Network latency varies. Request A fires, then B fires. B's response arrives before A's.
**How to avoid:** Use an `AbortController` to cancel the previous request when a new one fires. Or track a request counter and ignore responses from older requests.
**Warning signs:** Commission number "jumps" or flickers between values.

### Pitfall 3: Editing Sales with Pending Status Change Requests
**What goes wrong:** A sale has a pending StatusChangeRequest AND a manager submits a SaleEditRequest. Approval of either could conflict.
**Why it happens:** Two parallel approval workflows on the same sale.
**How to avoid:** Prevent creating a SaleEditRequest when a StatusChangeRequest is PENDING (and vice versa). Show a message like "This sale has a pending status change -- please wait for it to be resolved before editing."
**Warning signs:** Orphaned or conflicting change requests.

### Pitfall 4: Finalized Period Adjustment Entry Calculation
**What goes wrong:** When editing a sale in a finalized period, the adjustment amount is calculated incorrectly -- either too much or too little is adjusted.
**Why it happens:** The old commission must be subtracted and new commission added in the next open period's entry.
**How to avoid:** Follow the `handleCommissionZeroing` pattern. The adjustment entry in the next open period should represent the delta: `newCommission - oldCommission`. If the old entry is in a finalized period, create a new PayrollEntry in the next OPEN period with `adjustmentAmount = newCommission - oldCommission`.
**Warning signs:** Agent's total pay for the corrected period is wrong.

### Pitfall 5: Inline Edit State Persistence Across Tab Switches
**What goes wrong:** User starts editing a sale, switches to another tab, comes back, and the edit state is lost (or worse, stale).
**Why it happens:** React state is preserved but the sale list may have been refreshed.
**How to avoid:** Clear `editingSaleId` when switching tabs. On tab return, let the user re-enter edit mode.
**Warning signs:** Edit form shows stale data after tab switch.

### Pitfall 6: Agent Reassignment in Edit Creates Wrong Payroll Entry
**What goes wrong:** A sale's agent is changed, but the payroll entry still references the old agent.
**Why it happens:** `upsertPayrollEntryForSale` uses `sale.agentId` which may point to the new agent, but the old payroll entry still exists under the old agent.
**How to avoid:** When agent changes, delete the old payroll entry for the old agent, then call `upsertPayrollEntryForSale` which creates a new one under the new agent. Handle within a transaction.
**Warning signs:** Both old and new agent show commission for the same sale.

## Open Questions

1. **SaleEditRequest vs Extended StatusChangeRequest**
   - What we know: StatusChangeRequest stores old/new SaleStatus enums. SaleEditRequest needs to store arbitrary field diffs as JSON.
   - What's unclear: Whether combining them in one model with a `type` discriminator is worth the added complexity.
   - Recommendation: Use a separate `SaleEditRequest` model. The data shapes are fundamentally different (enum pair vs JSON diff). Separate models keep queries simpler and avoid null-field confusion. The payroll dashboard already groups by agentId -- it can fetch both types and merge them for display.

2. **GET /api/sales/:id for full sale details**
   - What we know: The current GET /api/sales list endpoint returns basic sale data with product/agent/leadSource includes.
   - What's unclear: Whether addon data is included in the list response (needed for inline edit pre-population).
   - Recommendation: Create or verify a GET /api/sales/:id single-sale endpoint that includes full addon data with product details. The inline edit form needs this to pre-populate addon checkboxes and premiums.

3. **Preview panel placement recommendation**
   - What we know: The form uses a two-column layout (form fields left, receipt parser + addons right).
   - What's unclear: Best placement for the commission preview panel.
   - Recommendation: Place the preview panel at the top of the right column, above the receipt parser. It is the most important feedback signal and should be immediately visible. When loading, show a subtle shimmer/pulse effect on the commission number. Keep previous value visible during loading to avoid layout shift.
