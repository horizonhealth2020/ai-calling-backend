# Phase 5: Commission Preview & Sale Editing - Context

**Gathered:** 2026-03-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Live commission preview on the sales form before submission, and full sale editing with approval workflow and commission recalculation. Preview uses server-authoritative calculation via a dedicated endpoint. Sale edits by managers create approval requests for payroll (same pattern as Phase 10 status changes). Covers SALE-05 and SALE-06.

</domain>

<decisions>
## Implementation Decisions

### Sale editing scope
- All sale fields are editable: product, premium, enrollment fee, payment type, addons, carrier, member info, dates, lead source, notes, AND agent (reassignment allowed)
- Editing happens inline in the agent sales tab on the manager dashboard (same pattern as status dropdown)
- MANAGER role can edit sales they entered; PAYROLL/SUPER_ADMIN can edit any sale directly (no approval needed)
- Manager edits create a pending change request — sale keeps original values until payroll approves
- Approval workflow reuses the Phase 10 StatusChangeRequest pattern (same DB model or parallel model)
- Payroll sees pending edit requests in the same "Pending Approvals" section of payroll cards alongside status change requests, distinguished by badge type

### Commission preview API
- Dedicated POST /api/sales/preview endpoint (not dry-run on existing create endpoint)
- Accepts the same payload as POST /api/sales
- Returns commission breakdown without creating anything in the database
- Uses the exact same `calculateCommission()` function as the real create/update flow — guarantees preview matches actual

### Preview update trigger
- Preview fires on product, addon, premium, enrollment fee, or payment type changes only
- Debounced ~500ms for premium and enrollment fee typing (numeric input fields)
- Does NOT fire on member name, notes, carrier, or other non-financial field changes

### Preview display
- Shows total commission at the top
- Collapsible or always-visible breakdown: core bundle amount, AD&D amount, bonus, any halving applied
- Subtle shimmer/pulse on the commission number while waiting for server response (keeps previous value visible)
- Breakdown helps managers understand why the number is what it is

### Edit recalculation UX
- When editing a sale, manager sees old-vs-new diff before submitting: "Commission: $45.00 → $90.00", "Period: 3/15-3/21 → 3/22-3/28"
- Payroll sees field-by-field diff when reviewing approval: "Premium: $50 → $75, Addon removed: Compass VAB, Commission: $45 → $90"
- The diff is generated by calling the preview endpoint with the new values and comparing against current stored values

### Finalized period edits
- Edits to sales in finalized periods ARE allowed
- Instead of modifying the finalized period's entry, create an adjustment entry in the next open period
- This preserves the finalized period's integrity while still allowing corrections

### Claude's Discretion
- Preview panel placement on the form (below submit, sidebar, or inline section)
- Whether to extend StatusChangeRequest model or create a separate SaleEditRequest model
- Exact debounce timing and loading animation implementation
- How inline editing activates (click row, edit button, double-click)
- Adjustment entry implementation details for finalized period edits

</decisions>

<specifics>
## Specific Ideas

- Approval workflow mirrors Phase 10 status change pattern — managers propose, payroll approves/rejects
- Commission preview should feel "live" — shimmer effect while recalculating, not a jarring spinner swap
- Field-by-field diff in payroll approval queue provides audit trail for all sale modifications
- Inline editing in agent sales tab keeps the workflow fast — no modals or page navigation

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `calculateCommission(sale)` in `payroll.ts`: Core calculation function — preview endpoint calls this directly
- `upsertPayrollEntryForSale(saleId)` in `payroll.ts`: Full recalc flow — used after approved edits
- `StatusChangeRequest` model from Phase 10: Pattern for approval workflow (pending/approved/rejected)
- `handleCommissionZeroing()` in `payroll.ts`: Handles finalized period adjustments (CLAWBACK_APPLIED pattern)
- Agent sales tab in `manager-dashboard/app/page.tsx`: Already has inline status dropdown — extend for full inline editing
- `authFetch()` from `@ops/auth/client`: Used for all API calls from dashboards

### Established Patterns
- Approval requests grouped by agentId in payroll cards (Phase 10 pattern)
- Amber/yellow left-border for pending approval sections (Phase 10 styling)
- Member ID shown for disambiguation in approval views (Phase 10 pattern)
- `window.confirm` for destructive actions (Phase 10 pattern)
- Inline `React.CSSProperties` constants for all styling
- Zod validation on all API endpoints

### Integration Points
- POST /api/sales/preview — new endpoint in `routes/index.ts`, reuses `calculateCommission()`
- PATCH /api/sales/:id — extend to allow product/addon/agent changes (currently limited to premium/fee)
- Payroll dashboard pending approvals section — extend to show sale edit requests alongside status changes
- Manager dashboard agent sales tab — add inline editing UI to existing sale rows

</code_context>
