<objective>
Close GAP-45-05. UAT Test 2 failed: locking period 03-29-2026 with FRONTED=200 created a `FRONTED HOLD=200` adjustment row on the SAME 03-29 period (labeled "Carried from prev week") instead of on the next period (04-05-2026). The 04-05 open week showed FRONTED=0 / HOLD=0.

**Root cause (diagnosed by reading carryover.ts L23 + payroll.ts getSundayWeekRange L12-L35):**

`executeCarryover` computes the next period id like this:
```ts
const nextDay = new Date(period.weekEnd.getTime() + 86400000);
const { weekStart, weekEnd } = getSundayWeekRange(nextDay);
const nextPeriodId = `${weekStart.toISOString()}_${weekEnd.toISOString()}`;
```

`period.weekEnd` is stored by `getSundayWeekRange` as a UTC midnight date for the Saturday of the source week (e.g. for week 03-29 → 04-04, weekEnd = `2026-04-04T00:00:00.000Z`). Adding 86400000 ms gives `2026-04-05T00:00:00.000Z` (Sunday UTC midnight).

`getSundayWeekRange` then runs `DateTime.fromJSDate(2026-04-05T00:00:00Z, { zone: 'America/New_York' })` which yields **Saturday 2026-04-04 20:00 EDT** (Eastern is UTC-4 in April). Luxon `weekday=6` (Saturday), so `daysSinceSunday = 6`, and `sunday = eastern.minus({days: 6}).startOf('day')` = `2026-03-29` — the SAME Sunday as the source period.

The function returns the SOURCE period's weekStart/weekEnd. `nextPeriodId === sourcePeriodId`. The upsert at L62 then targets the source period's `(agentId, payrollPeriodId)` composite key, creating/updating an adjustment row on the SAME period. UI labels it "Carried from prev week" because `holdFromCarryover=true` and `carryoverSourcePeriodId=periodId`.

Existing CARRY-01..CARRY-10 tests did NOT catch this because they mock `prisma.payrollPeriod.findUnique` and `prisma.agentPeriodAdjustment.upsert` directly, never exercising the real `getSundayWeekRange` arithmetic against actual UTC-stored period dates.

**Fix:** Compute `nextDay` from `period.weekStart` (the Sunday) plus 7 days 12 hours. weekStart is stored as Sunday UTC midnight. +7d 12h = next Sunday UTC noon. In Eastern that becomes next Sunday 8 AM, weekday=7, daysSinceSunday=0, sunday=that next Sunday — guaranteed correct regardless of DST and EDT/EST.

reverseCarryover does NOT need a code change: it queries `where: { carryoverSourcePeriodId: sourcePeriodId, holdFromCarryover: true }`, which finds whichever rows the (now-fixed) executeCarryover created. After this fix those rows live on the next period; reverseCarryover will still find them by the source-period back-pointer.

Purpose: Restore Path B (first-lock) so the carryover lands on the correct period. Re-locks via Path A then work because reverseCarryover has been correct since 45-02.

Output: One line change in carryover.ts, one new regression test that uses concrete UTC dates derived from getSundayWeekRange to prove next-period selection works across the timezone boundary.

<!-- Source-of-truth: getSundayWeekRange from apps/ops-api/src/services/payroll.ts L12-L35 (verbatim): -->
```ts
export const getSundayWeekRange = (date: Date, shiftWeeks: number = 0) => {
  const eastern = DateTime.fromJSDate(date, { zone: TIMEZONE });  // TIMEZONE = 'America/New_York'
  const daysSinceSunday = eastern.weekday === 7 ? 0 : eastern.weekday;
  const sunday = eastern.minus({ days: daysSinceSunday }).startOf('day');
  const shiftedSunday = shiftWeeks > 0 ? sunday.plus({ weeks: shiftWeeks }) : sunday;
  const saturday = shiftedSunday.plus({ days: 6 });
  const weekStart = new Date(Date.UTC(shiftedSunday.year, shiftedSunday.month - 1, shiftedSunday.day));
  const weekEnd = new Date(Date.UTC(saturday.year, saturday.month - 1, saturday.day));
  return { weekStart, weekEnd };
};
```

**Why `period.weekStart + 7d12h` works and `period.weekEnd + 1d` does not:**
- `weekStart` is stored as `2026-03-29T00:00:00.000Z` (UTC midnight Sunday).
- `+ (7 * 24 + 12) * 3600 * 1000` ms = `2026-04-05T12:00:00.000Z` (next Sunday UTC noon).
- In Eastern: `2026-04-05T08:00:00-04:00` (next Sunday 8 AM EDT). `eastern.weekday === 7`, so `daysSinceSunday = 0`, `sunday = 2026-04-05`. Correct next period.
- `weekEnd` is stored as `2026-04-04T00:00:00.000Z` (UTC midnight Saturday).
- `+ 86400000` ms = `2026-04-05T00:00:00.000Z` (UTC midnight Sunday).
- In Eastern: `2026-04-04T20:00:00-04:00` (Saturday 8 PM EDT). `eastern.weekday === 6`, `daysSinceSunday = 6`, `sunday = 2026-03-29`. **Wrong — same as source week.**

The 12-hour offset is intentional: it absorbs both EDT (UTC-4) and EST (UTC-5) plus a safety margin so the result is always firmly inside the next week's day window after timezone conversion.
