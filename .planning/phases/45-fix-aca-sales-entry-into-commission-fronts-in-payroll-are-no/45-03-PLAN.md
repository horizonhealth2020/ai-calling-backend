<objective>
Fix Bug 3: CS Submissions round robin cursor advances on page refresh, paste, and rep-list change instead of only on actual submission, causing uneven assignment. Research confirmed `batchRoundRobinAssign` in `repSync.ts:117-151` always persists the advanced cursor inside its own transaction, and `fetchBatchAssign` in `CSSubmissions.tsx:433-447` is called on every paste and every rep useEffect.

Fix shape: split preview vs commit server-side. Make `batchRoundRobinAssign` accept `{ persist, tx }` options. Client calls with `?preview=true` for all non-submit flows. The chargebacks and pending-terms POST handlers wrap `createMany` + cursor advance in a single `prisma.$transaction` so failed submissions roll back the cursor.

Purpose: Assignments stay fair even when payroll pastes, refreshes, or changes reps — only actual persisted submissions consume a cursor slot.

Output: Refactored helper, updated routes, client preview flag, new test file.
</objective>

    Change the signature and implementation to:
    ```ts
    import type { Prisma } from "@prisma/client";

    export async function batchRoundRobinAssign(
      type: "chargeback" | "pending_term",
      count: number,
      opts: { persist?: boolean; tx?: Prisma.TransactionClient } = {},
    ): Promise<string[]> {
      const persist = opts.persist !== false; // default true
      const settingKey = type === "chargeback"
        ? "cs_round_robin_chargeback_index"
        : "cs_round_robin_pending_term_index";

      const run = async (client: Prisma.TransactionClient | typeof prisma) => {
        const activeReps = await client.csRepRoster.findMany({ where: { active: true }, orderBy: { name: "asc" } });
        if (activeReps.length === 0) return [];

        const setting = await client.salesBoardSetting.findUnique({ where: { key: settingKey } });
        let idx = setting ? parseInt(setting.value, 10) : 0;
        if (Number.isNaN(idx) || idx < 0) idx = 0;

        const assignments: string[] = [];
        for (let i = 0; i < count; i++) {
          assignments.push(activeReps[idx % activeReps.length].name);
          idx++;
        }

        if (persist) {
          await client.salesBoardSetting.upsert({
            where: { key: settingKey },
            update: { value: String(idx % activeReps.length) },
            create: { key: settingKey, value: String(idx % activeReps.length) },
          });
        }

        return assignments;
      };

      if (opts.tx) {
        return run(opts.tx);
      }
      return prisma.$transaction((tx) => run(tx));
    }
    ```

    Preserve the existing `activeReps` lookup ordering and the `idx % activeReps.length` wrap-around semantics exactly. Do not rename the setting keys. Do not change the return type.
