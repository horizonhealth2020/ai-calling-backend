<task type="auto">
  <name>Task 1: Status change endpoint and commission zeroing</name>
  <files>apps/ops-api/src/services/payroll.ts, apps/ops-api/src/routes/index.ts</files>
  <action>
1. In apps/ops-api/src/services/payroll.ts, add a new exported function `handleCommissionZeroing(saleId: string)`:
   - Fetch all payroll entries for the sale, including their payrollPeriod.
   - For each entry: if `payrollPeriod.status === 'OPEN'`, zero it out: update payoutAmount to 0, netAmount to 0, status to 'ZEROED_OUT'.
   - If `payrollPeriod.status !== 'OPEN'` (finalized/locked/paid), apply clawback pattern: update adjustmentAmount to `Number(entry.adjustmentAmount) - Number(entry.netAmount)`, set status to 'CLAWBACK_APPLIED'. This mirrors the existing clawback logic at routes/index.ts lines 537-543.

2. In apps/ops-api/src/routes/index.ts, add PATCH /api/sales/:id/status endpoint (place it after the existing PATCH /sales/:id route):
   - Protected by requireAuth + requireRole("MANAGER", "SUPER_ADMIN")
   - Zod schema: `{ status: z.enum(["RAN", "DECLINED", "DEAD"]) }`
   - Fetch the sale. Return 404 if not found.
   - Use a Prisma transaction to prevent race conditions (per pitfall 2 in research):
     a. If oldStatus is DEAD or DECLINED and newStatus is RAN:
        - Check for existing PENDING change request for this sale. If one exists, return 409 with error "A pending change request already exists for this sale".
        - Create StatusChangeRequest with saleId, requestedBy (req.user.id), oldStatus, newStatus.
        - Log audit: "REQUEST_STATUS_CHANGE"
        - Return 200 with { changeRequest, message: "Change request created for payroll approval" }
     b. If oldStatus is RAN and newStatus is DEAD or DECLINED:
        - Update sale.status to newStatus directly.
        - Call handleCommissionZeroing(sale.id) to zero commission and handle finalized periods.
        - Log audit: "UPDATE_STATUS" with { oldStatus, newStatus }
        - Return updated sale.
     c. If Dead<->Declined (both non-RAN):
        - Update sale.status directly. No commission impact.
        - Log audit: "UPDATE_STATUS"
        - Return updated sale.
     d. If oldStatus === newStatus: return 200 with current sale (no-op).

3. Also add check: if sale has a PENDING change request when attempting Ran->Dead/Declined, cancel the pending request (set status to REJECTED, reviewedBy to current user) before applying the immediate status change. This prevents orphaned pending requests.

3. Add POST /api/status-change-requests/:id/reject endpoint:

   - Log audit: "REJECT_STATUS_CHANGE" with changeRequestId, saleId
