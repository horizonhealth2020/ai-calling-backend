  <action>
1. Fetch pending change requests:
   - Add a new state variable for pending change requests.
   - Fetch from GET /api/status-change-requests?status=PENDING using authFetch on component mount (alongside existing data fetches).
   - Group the pending requests by agentId (from request.sale.agentId) so they can be displayed in the correct agent's payroll card.

2. Display pending requests inside agent payroll cards:
   - Within each agent's payroll card, after the existing payroll entries list, add a "Pending Approvals" section if there are any pending requests for that agent.
   - Style the section with an amber/yellow left border or highlight using inline CSSProperties, following the dark glassmorphism theme. Example: `{ borderLeft: '3px solid #f59e0b', backgroundColor: 'rgba(245, 158, 11, 0.08)', padding: '12px', borderRadius: '8px', marginTop: '8px' }`.
   - Each pending request shows: sale member name, product, old status -> new status (e.g., "Dead -> Ran"), requested by (user name), requested at (formatted date).
   - Each pending request has two buttons: "Approve" (green) and "Reject" (red), styled as small inline buttons following the existing BTN CSSProperties pattern.

3. Wire approve/reject buttons:
   - Approve button calls POST /api/status-change-requests/:id/approve via authFetch.
   - Reject button calls POST /api/status-change-requests/:id/reject via authFetch.
   - On success, remove the request from local state and refetch payroll data (since approval triggers commission recalculation that changes payroll entries).
   - On error, show the error message.
   - Add loading state to buttons to prevent double-clicks.

4. Fix period totals to exclude $0 entries from Dead/Declined sales:
   - RESEARCH identified "at least 10 separate .reduce() calls" in the payroll dashboard that aggregate all entries. Each of these needs to filter by sale status.
   - For every .reduce() call that computes period summary totals (sale count, net amount, payout total), add a filter: only include entries where the associated sale's status is 'RAN' (or equivalently, where payoutAmount > 0, but status check is more correct since a RAN sale could theoretically have $0 payout).
   - The safest filter: check if the payroll entry's associated sale status is RAN. If the GET /payroll-entries response includes sale.status, filter on that. If not, filter on `entry.status !== 'ZEROED_OUT'` as a proxy (since Dead/Declined entries get ZEROED_OUT status from handleCommissionZeroing).
   - Review ALL .reduce() and aggregation code in the file. Do NOT miss any — the pitfall is specifically about forgetting some aggregation points.
   - $0 entries (ZEROED_OUT / Dead/Declined) should still render as rows in the agent card (visible for reporting) but with a muted/grayed-out style to distinguish them from active entries.

5. Add a StatusBadge-like display for payroll entries that shows the sale status (Ran/Declined/Dead) next to each entry. Use the same color scheme: RAN=green, DECLINED=red, DEAD=gray. This gives payroll staff visibility into why an entry is $0.
