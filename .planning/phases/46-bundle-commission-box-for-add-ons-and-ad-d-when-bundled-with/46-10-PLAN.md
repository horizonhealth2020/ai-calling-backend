<objective>
GAP-46-UAT-05 (round 2): The CS → payroll chargeback alert pipeline is reaching `createAlertFromChargeback` correctly for MATCHED rows (verified by GAP-46-UAT-01 / Plan 46-06 observability), but rounds-1+2 of UAT confirmed via the 46-06 diagnosis comment that UNMATCHED and MULTIPLE chargebacks are silently dropped from `alertPayloads` at chargebacks.ts:204 (`if (cb.matchStatus !== "MATCHED" || !cb.matchedSaleId) continue;`). In production this is the dominant failure mode — typos, formatting drift, leading-zero memberIds, and members with multiple active sales all skip the alert pipeline entirely.

User chose **Option B (durable fix)**: allow CS to queue alerts for UNMATCHED/MULTIPLE rows too, carrying the raw memberId/memberName so payroll can manually pick the correct sale during approval. The schema is already friendly — `PayrollAlert` does NOT have a `saleId` column at all; the sale link flows via `payrollAlert.chargebackSubmissionId → chargebackSubmission.matchedSaleId`, which is already nullable. **No schema migration is needed.** The fix is purely application-layer: relax the MATCHED gate, surface raw identity, add a manual sale-picker in the approve UI, and have approveAlert write the picked sale onto the chargeback row before running the existing clawback path.

Purpose: Close the last UAT gap on Phase 46. CS users get a single-step submit; payroll gets a clear "this needs your attention" row in the Chargebacks container with all the info needed to disambiguate; the existing approveAlert dedupe + clawback logic is reused without rewrite.

Output: Three coordinated edits — server (alert payload widening + approveAlert manual-pick branch + alerts route), payroll dashboard (UI sale picker on unmatched rows + Alert type widening), CS dashboard (toast transparency).
