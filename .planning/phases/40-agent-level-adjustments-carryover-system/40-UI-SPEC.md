## Interaction Contract

### EditableLabel (NEW)

**Purpose:** Allow payroll staff to override auto-generated labels (e.g., "Hold Payout" to "Bonus", or "Fronted Hold" to "Hold").

**States:**
1. **Display (default):** Renders as `<span>` with HEADER_LBL style. Cursor changes to `pointer` on hover.
2. **Editing:** Renders as `<input>` with SMALL_INP base style at width 100px, fontSize 11px. Auto-focuses on mount.
3. **Saving:** On blur or Enter key, calls `onChange(draft)` and returns to display state.
4. **Cancel:** On Escape key, reverts to original value and returns to display state.

**Visual spec:**
- Display state: `{ fontSize: 11, fontWeight: 700, textTransform: "uppercase", letterSpacing: "0.06em", cursor: "pointer" }`
- Color when from carryover (bonus): `colors.success` (#34d399)
- Color when from carryover (hold): `colors.warning` (#fbbf24)
- Color when default label: `colors.textMuted` (#475569) -- matches existing HEADER_LBL
- Edit input: `{ ...SMALL_INP, width: 100, fontSize: 11, padding: "2px 8px" }`
- Transition: none (instant swap between display and edit)

**Accessibility:**
- Display span gets `role="button"` and `tabIndex={0}`
- Enter and Space on display state trigger edit mode
- Edit input traps focus until blur/Enter/Escape

### CarryoverHint (NEW)

**Purpose:** Show "Carried from prev week" below bonus or hold inputs when the value was auto-populated by carryover.

**Rendering condition:** Show when `bonusFromCarryover === true` (below bonus input) or `holdFromCarryover === true` (below hold input).

**Visual spec:**
- Text: "Carried from prev week"
- Style: `{ fontSize: 11, fontWeight: 400, color: colors.textMuted, fontStyle: "italic", marginTop: 4, lineHeight: "1.45" }`
- No icon, no background -- pure text annotation

### Approve/Unapprove Buttons (MODIFIED)

**Visibility rules (changed):**
- Show "Approve" when: `entry.halvingReason` is truthy AND `entry.sale?.commissionApproved` is false
- Show "Unapprove" when: `entry.halvingReason` is truthy AND `entry.sale?.commissionApproved` is true
- Hide both when: `entry.halvingReason` is falsy

**No visual style changes** -- only the conditional logic changes.

### Print Pills (MODIFIED)

**Position change:** Pills move from AFTER commission amount to BEFORE commission amount.

**Before:** `$125.00 [Approved]`
**After:** `[Approved] $125.00`

**HTML pattern:**
```html
<td class="right" style="font-weight:700">${commFlagHtml}$${amount}</td>
```

Pills render as `inline-block` with existing styles:
- Approved: `{ background: "rgba(52,211,153,0.12)", color: "#34d399", fontSize: 11, fontWeight: 700, borderRadius: 4, padding: "1px 8px", marginRight: 4 }`
- Half commission: `{ background: "rgba(251,191,36,0.12)", color: "#fbbf24", fontSize: 11, fontWeight: 700, borderRadius: 4, padding: "1px 8px", marginRight: 4 }`

The `marginRight: 4` ensures a 4px gap between the pill and the dollar amount.

### Zero-Sales Agent Card (MODIFIED)

**New behavior:** Agent cards render even when `entries.length === 0`, as long as an `AgentPeriodAdjustment` record exists for that agent+period.

**Visual when zero sales:**
- Agent name header: renders normally
- Sale count text: "0 sales"
- Commission display: "$0.00"
- Bonus/Fronted/Hold inputs: functional, populated from AgentPeriodAdjustment values
- Net amount: computed as `0 + bonus + fronted - hold` (may be negative)
- Negative net display: `{ color: colors.danger }` (#f87171)
- Sale table: hidden (no rows to show)
- Print button: still visible (prints the adjustment summary)

### Empty States

| Context | Copy |
|---------|------|
| Agent card with zero sales + carryover | Card renders with "$0.00" commission and populated adjustment inputs. No empty state message needed -- the inputs ARE the content. |
| Agent card with zero sales + no carryover | Card does NOT render (no change from current behavior) |

### Error States

| Context | Copy |
|---------|------|
| Label save failure | Input reverts to previous value silently (inline edit pattern -- no toast) |
| Adjustment save failure | Input reverts; status bar shows "Request failed ({status})" using existing error pattern |
| Carryover execution failure on lock | Lock action fails; existing error handling displays "Request failed ({status})" |

### Destructive Actions

| Action | Confirmation |
|--------|-------------|
| Unapprove (reverts to half commission) | No confirmation dialog -- single click (existing pattern, unchanged) |
| Period lock (triggers carryover) | No additional confirmation beyond existing lock flow. Carryover is automatic and non-destructive (adds to next period). |
