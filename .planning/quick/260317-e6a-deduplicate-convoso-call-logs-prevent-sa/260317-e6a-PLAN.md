<objective>
Deduplicate Convoso call log processing so the KPI poller does not recount the same calls across poll cycles.

Purpose: The poller runs every 10 minutes and fetches ALL call logs each time. Without deduplication, AgentCallKpi rows accumulate duplicates and call counts inflate. The CallAudit auto-tag in POST /sales is already idempotent (updateMany sets status to "sale_matched" which is a no-op on repeat), so the primary fix targets the KPI poller.

Output: ProcessedConvosoCall tracking table, deduplicated poller, automatic 30-day cleanup.
</objective>

1. Add a new `ProcessedConvosoCall` model to `prisma/schema.prisma`:

```prisma
model ProcessedConvosoCall {
  id              String   @id @default(cuid())
  convosoCallId   String   @unique @map("convoso_call_id")
  leadSourceId    String?  @map("lead_source_id")
  processedAt     DateTime @default(now()) @map("processed_at")

  @@index([processedAt])
  @@map("processed_convoso_calls")
}
```

Fields:
- `convosoCallId`: The unique `id` from the Convoso API response. Unique constraint prevents duplicates.
- `leadSourceId`: Optional reference for debugging which lead source the call came from.
- `processedAt`: Timestamp for 30-day cleanup queries. Indexed for efficient deletion.

No foreign key to LeadSource (the Convoso call ID is the source of truth, and we want minimal coupling).

2. Create manual migration SQL at `prisma/manual-migrations/add_processed_convoso_call.sql`:

```sql
CREATE TABLE IF NOT EXISTS processed_convoso_calls (
  id TEXT PRIMARY KEY,
  convoso_call_id TEXT NOT NULL UNIQUE,
  lead_source_id TEXT,
  processed_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_processed_convoso_calls_processed_at ON processed_convoso_calls (processed_at);
```

1. **In `pollLeadSource`**, after `extractConvosoResults(response)` and before `enrichWithTiers`:

   a. Extract all Convoso `id` values from the raw results: `const callIds = raw.map((r: any) => String(r.id)).filter(Boolean)`

   b. Query existing processed IDs in a single batch:
   ```typescript
   const existing = await prisma.processedConvosoCall.findMany({
     where: { convosoCallId: { in: callIds } },
     select: { convosoCallId: true },
   });
   const existingSet = new Set(existing.map(e => e.convosoCallId));
   ```

   c. Filter raw results to only new calls:
   ```typescript
   const newRaw = raw.filter((r: any) => !existingSet.has(String(r.id)));
   ```

   d. Log the dedup stats:
   ```typescript
   if (existingSet.size > 0) {
     console.log(JSON.stringify({
       event: "kpi_poll_dedup",
       leadSourceId: leadSource.id,
       totalFetched: raw.length,
       alreadyProcessed: existingSet.size,
       newCalls: newRaw.length,
       timestamp: new Date().toISOString(),
     }));
   }
   ```

   e. If `newRaw.length === 0`, return 0 early (all calls already processed).

   f. Use `newRaw` instead of `raw` for `enrichWithTiers(newRaw)` and the rest of the KPI logic.

   g. After the successful `prisma.agentCallKpi.createMany`, batch-insert the new call IDs:
   ```typescript
   const newCallIds = newRaw.map((r: any) => String(r.id)).filter(Boolean);
   if (newCallIds.length > 0) {
     await prisma.processedConvosoCall.createMany({
       data: newCallIds.map(cid => ({
         convosoCallId: cid,
         leadSourceId: leadSource.id,
       })),
       skipDuplicates: true,
     });
   }
   ```

2. **In `runPollCycle`**, after the lead source polling loop completes (after the `for` loop, before the final log), add 30-day cleanup:

   ```typescript
   // Cleanup processed call records older than 30 days
   try {
     const cutoff = new Date(Date.now() - 30 * 24 * 60 * 60 * 1000);
     const { count } = await prisma.processedConvosoCall.deleteMany({
       where: { processedAt: { lt: cutoff } },
     });
     if (count > 0) {
       console.log(JSON.stringify({
         event: "kpi_poll_cleanup",
         deletedRecords: count,
         cutoffDate: cutoff.toISOString(),
         timestamp: new Date().toISOString(),
       }));
     }
   } catch (cleanupErr: any) {
     console.error(JSON.stringify({
       event: "kpi_poll_cleanup_error",
       error: cleanupErr.message,
       timestamp: new Date().toISOString(),
     }));
   }
   ```

Important: The full KPI summary (enrichWithTiers + buildKpiSummary + AgentCallKpi createMany) should still run on the new calls only. The existing AgentCallKpi rows from previous polls remain as historical snapshots — this is correct behavior since each snapshot captures the state at poll time.
