  <action>
    Systematically audit and remove unused imports across every source file in the monorepo.

    APPROACH -- Work app by app, then packages:

    1. **ops-api** (41 files): For each .ts file in apps/ops-api/src/, check every import statement. If the imported symbol is not referenced anywhere else in that file (not in function bodies, type annotations, or re-exports), remove the import line. Use TypeScript compiler as a helper:
       ```
       cd apps/ops-api && npx tsc --noEmit --noUnusedLocals 2>&1 | grep "declared but"
       ```
       This will flag unused local imports in TypeScript files. Fix each flagged import.

    2. **ops-dashboard** (38 files): Same approach for all .ts/.tsx files under apps/ops-dashboard/. Run:
       ```
       cd apps/ops-dashboard && npx tsc --noEmit --noUnusedLocals 2>&1 | grep "declared but"
       ```

    3. **sales-board** (5 files): Same approach for apps/sales-board/.

    4. **Morgan** (7 JS files): Morgan uses CommonJS require(). TypeScript compiler will not check these. For each .js file in apps/morgan/, manually read the file and verify every require() and destructured import is actually used in the file. Remove any that are not.

    5. **Shared packages** (packages/auth, packages/db, packages/types, packages/ui, packages/socket, packages/utils): For each .ts/.tsx file, check imports. Note: barrel file re-exports (like `export { X } from './Y'`) are NOT unused imports even if X is not used in that file -- they are re-exports for consumers.

    SAFETY RULES:
    - Do NOT remove imports that are used only as types (e.g., `import type { X }` where X is used in a type annotation)
    - Do NOT remove side-effect imports (e.g., `import './styles.css'` or `import 'dotenv/config'`)
    - Do NOT remove imports in barrel/index files that re-export (these are used by consumers)
    - After all removals, run: `npm test && npm run test:ops` to verify nothing broke
    - If any test fails, revert the last change and investigate

    COMMIT after all import cleanup: one commit with message describing the scope.

  </files>
  <action>
    Scan every source file for commented-out code blocks and remove them. Preserve explanatory comments.

    WHAT TO REMOVE (commented-out code):
    - Lines starting with `//` that contain code patterns: variable declarations (`const`, `let`, `var`), function calls, return statements, import/export statements, JSX tags (`<`), assignments (`=`), await expressions, if/for/while statements
    - Multi-line comment blocks (`/* ... */`) that contain disabled code (function bodies, JSX, logic)
    - Blocks of consecutive commented lines that form a coherent code block (e.g., a commented-out function)

    WHAT TO KEEP (explanatory comments):
    - Comments explaining WHY something is done (e.g., `// SUPER_ADMIN bypasses role checks intentionally`)
    - TODO/FIXME/NOTE comments
    - JSDoc comments and type documentation
    - Single-line comments explaining the next line of code
    - License headers
    - Comments marking section boundaries (e.g., `// === PAYROLL ROUTES ===`)

    APPROACH:
    1. Search for commented-out code patterns:
       ```
       grep -rn "^\s*//\s*\(const \|let \|var \|return \|import \|export \|await \|if (\|for (\|while (\|<[A-Z]\)" apps/ packages/ --include="*.ts" --include="*.tsx" --include="*.js" | grep -v node_modules | grep -v ".next"
       ```
    2. For each match, read the surrounding context to determine if it is disabled code or an explanatory comment.
    3. Remove confirmed commented-out code blocks.
    4. Also scan for multi-line `/* */` blocks containing code.

    SAFETY: If unsure whether a comment is explanatory or disabled code, leave it. Per user decision D-05: "Comments explaining 'why' are fine; disabled code is not." Per user decision D-03: "Do not affect how the system operates."

    After all removals, run tests to verify nothing broke (comments should never affect runtime, but verify anyway per D-08).
  </action>

  </verify>
